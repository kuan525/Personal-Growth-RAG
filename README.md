## 建议的核心路线

放弃“先把理论补齐再开始”的路线，改为：

**先做一个能用的本地文档问答助手，再在迭代中补知识。**

这是最适合你当前时间预算和学习偏好的路径。

## 最适合你的实战项目

### 项目题目

本地文档问答助手

### 最小目标

输入：PDF / Markdown / TXT 文档  
输出：答案 + 引用原文片段

### 第一阶段只做 4 个核心模块

1. 文档读取
2. 文档分块（chunking）
3. 向量化（embedding）
4. 检索后交给 LLM 生成回答

### 第一阶段成果标准

不是“代码优雅”或“架构完整”，而是：

- 能导入你自己的文档
- 能提出真实问题
- 能返回比较靠谱的答案
- 能展示命中的原文片段或引用

## 建议的学习顺序

### 第 1 步：先做最小可用版

先基于你看过的 Local_Pdf_Chat_RAG 一类结构做一个 baseline，不要自己重新设计复杂架构。

先做到：

- 支持导入本地文档
- 建索引
- 查询
- 返回答案和引用

### 第 2 步：立刻用真实问题测试

不要只问演示型问题，要直接拿自己的真实资料测试，准备 15~20 个问题，例如：

- 文档里明确写过的问题
- 需要跨多段信息组合的问题
- 容易混淆的术语问题
- 表格、图片附近的信息问题

### 第 3 步：按问题补知识，而不是按目录学知识

遇到什么问题，就补对应知识：

- 答非所问 → 学 chunking、top_k、prompt 约束
- 文档里有但检索不到 → 学 BM25、hybrid retrieval、query rewrite
- PDF 解析混乱 → 学 OCR、layout analysis、结构化切块
- 回答看起来通顺但不可信 → 学引用、faithfulness、基于上下文作答约束
- 文档多了以后更新或性能有问题 → 学向量数据库服务化方案

## 具体技术路线建议

### 向量数据库

现阶段不要一开始就重投入学习复杂向量数据库。

建议顺序：

- 先用 FAISS 跑通最小系统
- 当你需要持久化、metadata filter、频繁更新、服务化时，再上 Qdrant
- 暂时不用优先深挖 Milvus、Pinecone 或分布式部署

### RAG 框架

现阶段不要同时学习多个框架。

建议：

- 优先沿着现有 demo 思路自己搭一个基础版本
- 后续如果需要更系统的抽象，可以补 LlamaIndex
- 暂时不把主要精力放在啃 RAGFlow 源码上

### 文档解析

文档解析是后续迭代的重要方向，尤其是复杂 PDF 的场景。

当你发现：

- 表格信息丢失
- 标题层级混乱
- OCR 不稳定
- 纯文本切块效果差

就重点补 chunkr 这类“文档解析/结构化前处理”方向。
