# 个人图谱设计

## 1. 设计目标

Personal graph 的目标是连接你的项目、技能、目标、决策、问题、资源和 evidence，让系统不只是检索片段，而是能理解个人知识之间的关系。

它不是可视化玩具，也不是企业级 knowledge graph。

## 2. 设计原则

只抽取能帮助真实问题的关系。

好的 graph questions：

- 哪些项目和 RAG 相关？
- 这个项目做过哪些 decision？
- 哪些 problem 反复出现？
- 这个项目锻炼了哪些 skill？
- 某个 decision 有哪些 evidence 支持？

不好的 graph 目标：

- 抽取所有名词
- 一开始设计巨大 ontology
- 只为了展示漂亮关系图

## 3. Entity Types

初始 entity types：

```text
Project
Skill
Topic
Goal
Decision
Problem
Resource
Evidence
Person
Event
```

## 4. Entity 定义

### Project

有明确目标和时间线的具体实践。

例子：

- Personal Growth RAG
- WeChat group report pipeline
- image generation skill

### Skill

用户正在学习或应用的能力。

例子：

- RAG
- retrieval evaluation
- document parsing
- project planning

### Goal

希望达成的结果。

例子：

- 构建个人 RAG
- 完成 MVP
- 提升 retrieval quality

### Decision

在多个 options 中做出的选择。

例子：

- 先用 FAISS，后续再考虑 Qdrant
- 先做 incremental update，再做 graph

### Problem

阻塞、反复出现的问题或质量风险。

例子：

- PDF parsing 差
- retrieval 找不到已知答案
- planning 太泛

### Resource

文档、笔记、文章、repo 或其他 source。

### Evidence

支持某个 claim 的具体 chunk 或 quote。

## 5. Relation Types

初始 relation types：

```text
PART_OF
RELATED_TO
SUPPORTS
CONTRADICTS
CAUSED_BY
LED_TO
NEXT_ACTION
REVIEWED_IN
DEPENDS_ON
```

## 6. Relation 定义

### PART_OF

某个 entity 属于另一个 entity。

例子：

- retrieval evaluation PART_OF Personal Growth RAG

### SUPPORTS

evidence 支持 decision、claim 或 memory。

例子：

- chunk_123 SUPPORTS decision_use_faiss_first

### CAUSED_BY

某个 problem 由某个原因导致。

例子：

- poor_answer CAUSED_BY retrieval_miss

### LED_TO

某个 decision 或 event 导致某个 outcome。

例子：

- choose_mvp_first LED_TO faster_feedback

### NEXT_ACTION

review 或 decision 产生后续行动。

例子：

- review_week_1 NEXT_ACTION build_validation_set

## 7. Evidence 要求

每条 relation 必须包含：

```text
evidence_chunk_id
confidence
created_at
status
```

没有 evidence 的 relation 不能 active。

## 8. 抽取策略

### Stage 1：Candidate Extraction

从以下内容中用 rules 或 LLM extraction 生成 candidate entities 和 relations：

- project docs
- review notes
- decision logs
- planning docs

### Stage 2：Evidence Linking

每个 candidate relation 必须关联一个或多个 chunks。

### Stage 3：Human Review

重要 relation 必须可检查、可修正。

### Stage 4：Retrieval Integration

graph relation 只有在改善真实问题时才参与 context expansion。

## 9. Graph-assisted Retrieval

graph expansion 适用于 query 涉及：

- project context
- decisions
- goals
- repeated problems
- skills
- time-based review

示例流程：

1. 识别 query 中的 entities。
2. 检索匹配 graph nodes。
3. 扩展 related nodes 和 evidence chunks。
4. 与 dense / sparse retrieval chunks 合并。
5. 对 final candidates rerank。

## 10. Correction Workflow

用户必须能：

- disable wrong entity
- merge duplicate entities
- disable wrong relation
- update entity aliases
- mark relation as low-confidence

## 11. 验收标准

Graph MVP 只有满足以下条件才算通过：

- relations 有 evidence
- graph entries 可检查
- graph 至少改善 3 个真实问题
- wrong entries 可修正
- graph noise 受控

## 12. 失败条件

graph design 出现以下情况即失败：

- 抽取很多 entity，但没有改善 answer
- relation 没有 evidence
- graph error 无法修正
- graph 只是装饰
- graph expansion 让 retrieval 噪音更大且不可控
