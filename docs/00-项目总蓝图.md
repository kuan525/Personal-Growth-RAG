# 项目总蓝图：个人成长型 RAG

## 1. 项目定位

这个项目不是普通 RAG demo，也不是一个简单的本地文档问答工具。

它的目标是构建一个 **个人成长与决策支持知识系统**。系统以 RAG 为基础，持续吸收你的文档、项目记录、复盘、决策、学习资料和经验沉淀，把这些内容组织成可检索的 evidence、结构化 metadata、个人知识图谱、长期 memory，并在未来的学习、项目推进、阶段复盘和关键决策中，为你提供有证据、有边界、可追溯的建议。

一句话定义：

> 这是一个本地优先、可增量更新、带个人图谱和长期 memory 的 RAG 系统，最终服务于个人实践、复盘和决策。

## 2. 产品本质

这个系统应该逐步成长为你的个人知识基础设施，而不是一次性工具。

它包含四层递进能力：

1. **Evidence Retrieval**：能从个人资料中检索证据，并带 citation 回答问题。
2. **Incremental Knowledge Base**：资料持续增长时，能安全地新增、修改、删除，不需要每次全量重建。
3. **Personal Knowledge Graph**：把项目、技能、目标、问题、决策、资源和 evidence 连接起来。
4. **Decision Support**：在具体实践或时间节点，基于历史记录和 evidence 给出建议。

产品承诺是：

> 当你面对一个项目、学习路径、复盘或决策时，系统能调出相关历史，展示支持证据，指出风险，并给出下一步行动建议。

## 3. North Star

项目成功的最高标准不是“功能很多”，而是你会自然地在真实场景中使用它。

典型场景包括：

- 开始新项目之前，问：我以前做过什么类似事情？当时踩过什么坑？
- 实现过程中，问：我为什么选择这个方向？下一步最小行动是什么？
- 每周复盘时，问：本周反复出现的问题是什么？
- 做规划时，问：我应该继续推进 RAG，还是切换到另一个方向？
- 学习时，问：哪些技能我真正实践过，而不只是看过资料？

## 4. 核心原则

### 4.1 Evidence before opinion

系统可以给建议，但建议必须建立在 evidence 上。

每个建议都应该区分：

- 检索到的 evidence
- 历史模式
- 模型推断
- 缺失信息
- 不确定性

### 4.2 Incremental by design

这个系统会长期增长，因此 document、chunk、embedding、index、graph relation、memory 都必须从设计上支持更新、失效和删除。

不能等到后期再补 incremental update，否则会推倒重来。

### 4.3 Graph 是工具，不是装饰

个人图谱不是为了画一张复杂关系图，而是为了回答更好的问题：

- 这个项目和哪些技能相关？
- 哪些问题反复出现？
- 某个决策有哪些 evidence 支持？
- 哪些目标没有推进？

### 4.4 Memory 必须可修正

个人 memory 一旦错误，会长期污染系统判断。

所以每条 memory、relation、fact 都必须有 source，并且可以编辑、废弃、失效或删除。

### 4.5 Local-first, single-user first

当前系统服务的是你自己。

优先级是：本地数据、本地运行、单用户、真实个人场景。

不优先考虑：

- 多用户
- 权限系统
- SaaS 化
- 企业后台
- 团队协作

## 5. 系统分层

系统整体分为七层：

1. **Source Layer**：文档、笔记、项目记录、复盘、决策记录。
2. **Ingestion Layer**：解析、规范化、chunking、metadata 提取。
3. **Storage Layer**：documents、chunks、embeddings、indexes、graph、memories、logs。
4. **Retrieval Layer**：dense retrieval、sparse retrieval、rerank、metadata filter、graph expansion。
5. **Graph Layer**：entities、relations、evidence links、mentions。
6. **Memory Layer**：目标、偏好、经验、项目状态、反复问题。
7. **Answer & Decision Layer**：带 citation 的回答、总结、建议、复盘和决策辅助。

## 6. 范围边界

### 6.1 当前范围内

- 本地文档 ingestion
- PDF / Markdown / TXT 支持
- Citation-based QA
- Incremental update
- Metadata 与 index 管理
- 轻量 personal graph
- Review / decision record ingestion
- Evidence-based decision support
- Evaluation 与 regression testing

### 6.2 当前范围外

- 多用户协作
- SaaS 产品化
- 企业权限系统
- 复杂 multi-agent automation
- 一开始就做完整 GraphRAG
- 完整 OCR / multimodal parsing
- 自动操作外部系统
- 没有来源的个人建议

## 7. 战略实现顺序

项目必须按下面顺序推进：

1. 先证明资料能被问答。
2. 再证明 retrieval 能稳定找到正确 evidence。
3. 再证明资料能 incremental update。
4. 再证明知识能通过 personal graph 连接。
5. 再证明 review 和 decision 能沉淀成 memory。
6. 再证明系统能辅助决策。
7. 最后证明质量能长期稳定。

不要跳阶段。后面的智能能力如果建立在不可靠的 evidence、update 和 traceability 上，价值会很低。
