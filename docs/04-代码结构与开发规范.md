# 代码结构与开发规范

## 目标

本项目已经完成最小 RAG MVP，后续重构和新增功能都应优先保持可读性、可追踪性和可复用性。代码组织不追求复杂企业架构，而是让人快速判断：入口在哪里、业务数据流在哪里、外部依赖在哪里、数据库在哪里、prompt 在哪里。

## 分层边界

```text
api / cli
  -> application
    -> domain
    -> infrastructure
  -> core / utils
```

### api/v1

HTTP 入口层，只做请求接收、依赖注入、调用 use case、响应转换和 HTTP 错误映射。API 层不直接写 SQL，不直接调用 DashScope / DeepSeek / FAISS，不构造复杂 prompt。请求/响应模型放在 `api/v1/contracts`，这些模型是对外契约，不和 DB model、domain data 混用。

### cli

命令行入口层，和 API 一样只做参数解析和 use case 调用。CLI 不应该复制 ingestion、retrieval、QA 逻辑；否则 API 和 CLI 会在后续演进中分叉。

### application

业务用例层，是项目主数据流所在地。文件名使用业务动作，例如 `ingest_document.py`、`search_chunks.py`、`answer_question.py`。application 可以编排 repository、provider、vectorstore，但不应该把 HTTP contract 当作内部数据结构。

### domain

轻量业务概念层，当前只放稳定 dataclass、prompt、状态语义和拒答文案。不要过早做复杂 DDD；当 memory、decision、graph 出现稳定概念后，再自然扩展。

### infrastructure

具体技术实现层：SQLite/SQLAlchemy、FAISS、DashScope、DeepSeek、pdfminer、LangChain splitter 都属于这里。repository 只处理数据库读写；provider 只处理外部模型调用；vectorstore 只处理 index 读写和 search。

### core / utils

`core` 放全局配置、日志、错误、路径、常量；不放业务数据流。`utils` 放无状态工具，例如 hash、id、time。

## 当前数据流

### 文档导入

```text
POST /v1/documents 或 CLI ingest_dir
  -> application.documents.ingest_document
  -> parse_document
  -> split_text
  -> documents/chunks 写入 SQLite
  -> DashScope embedding
  -> FAISS append
  -> embeddings 写入 SQLite
  -> document 标记 active
```

设计取舍：当前先将 document 落为 `processing`，失败时写 `failed + error_message`，这样解析失败不会静默丢失。FAISS 目前是 append-only 派生索引，后续应补 `rebuild_index` 来处理删除、更新和一致性校验。

### 语义检索

```text
POST /v1/search
  -> application.retrieval.search_chunks
  -> query embedding
  -> FAISS search
  -> embeddings.index_position 反查 chunk_id
  -> SQLite hydrate chunk/document
  -> 过滤非 active document
  -> 返回 evidence chunks
```

设计取舍：检索结果最终以 SQLite 为准。FAISS 命中但 SQLite 找不到 metadata 时跳过，这能容忍短期索引不一致，也为后续 rebuild 留出空间。

### RAG 问答

```text
POST /v1/questions
  -> application.questions.answer_question
  -> retrieval.search_chunks
  -> build_question_prompt
  -> DeepSeekLLMClient
  -> answer + citations
```

设计取舍：没有 evidence 时直接返回“现有资料不足以回答这个问题”，不让 LLM 自行补全。citations 当前表示“用于回答的候选 evidence”，下一步 Question Trace 会把 question、retrieval results、answer、citations 落库。


## API 字段边界

统一响应 envelope 负责表达请求成功或失败：`success/error_code/message/data/request_id`。成功态 `data` 只放产品需要的业务字段，不放内部排障字段。比如 Document API 不返回 `stored_path/chunk_path/error_message`；这些字段仍保存在 SQLite，供本地排障、CLI 或未来 admin/debug 接口使用。

## 注释规范

注释可以适当存在，但只解释高价值信息：为什么这样做、当前阶段的取舍、后续扩展边界、数据一致性假设。不要解释显而易见的代码行为。

推荐：

```python
# FAISS 是 SQLite metadata 的派生索引；后续增量删除和 compact 应通过 rebuild 解决。
```

不推荐：

```python
# 保存 index 文件
```

## 新增功能落点

- Question Trace：`application/questions` 编排落库，DB 表和 repository 放 `infrastructure/db`，回放接口放 `api/v1/routes`。
- Retrieval Quality：先扩展 `application/retrieval`，再增加 hybrid/rerank provider，不要塞进 API route。
- Index Rebuild：放 `application/indexing` 或 `cli/rebuild_index.py`，SQLite 为事实源，FAISS 可重建。
- Evaluation：放 `application/evaluation`，复用 retrieval/questions，不走 HTTP。
- Memory / Decision Support：稳定概念放 `domain`，流程放 `application`，证据链接仍回到 chunk/document。
