# 路线图与阶段范围

## 1. 文档目的

本文件用于定义整个系统的分期建设路线，明确每一阶段解决什么问题、为什么此时做、交付什么能力、如何判断这一阶段已经完成。

这个文档的目标不是列一个理想化大清单，而是给出一条真实可执行的建设路径。

## 2. 整体分期原则

整个系统按以下逻辑推进：

1. 先做出能用的最小闭环
2. 再解决检索质量问题
3. 再解决输入数据质量问题
4. 再解决反复使用时的操作成本问题
5. 最后建立长期迭代所需的评测和稳态机制

每一期都必须满足三个条件：

- 本期结束后，用户获得了新的实际能力
- 本期可以独立验证是否完成
- 本期不依赖后续阶段才能证明价值

## 3. 阶段总览

| 阶段 | 名称 | 核心目标 | 交付结果 | 验证方式 |
| --- | --- | --- | --- | --- |
| Phase 1 | MVP 本地 RAG 基线 | 先把完整问答链路跑通 | 本地文档可问答、可引用 | 真实问题集验证 |
| Phase 2 | 检索质量提升 | 提升命中率和上下文质量 | hybrid retrieval + rerank | 与 Phase 1 对比测试 |
| Phase 3 | 文档解析与进入系统能力 | 提升复杂文档的可用性 | 更好的解析、结构化 chunk 和 metadata | 困难文档测试 |
| Phase 4 | 可用性与运营能力 | 让系统可以稳定反复使用 | index 管理、配置、状态与日志 | 操作流程测试 |
| Phase 5 | 评测与稳态建设 | 建立长期迭代和回归机制 | 评测集、回归流程、发布检查 | 重复评测与对比 |

## 4. Phase 1：MVP 本地 RAG 基线

### 4.1 阶段目标

做出一个用户可以真正开始使用的本地文档问答系统。

### 4.2 本期解决的问题

在这个阶段之前，用户只有概念和参考项目，没有自己的可运行系统。

本期要解决：

- 文档能否被导入和索引
- 用户能否提问并获得答案
- 回答是否能附带引用
- 这个产品是否具备最小可用价值

### 4.3 本期交付能力

- 支持本地导入 PDF / Markdown / TXT
- 支持文本抽取与切块
- 支持 embedding 和本地索引
- 支持提问、检索、回答
- 支持基本引用展示
- 支持一组真实问题验证

### 4.4 本期不做

- hybrid retrieval
- rerank
- 高级 PDF 结构解析
- 大规模 corpus 管理
- 自动化评测体系
- 企业级部署能力

### 4.5 完成标准

如果用户可以拿自己的资料做基本问答，并且多数简单问题能答对且附带引用，就认为本期完成。

### 4.6 进入下一期条件

只有当系统已经具备最小可用价值，才有必要投入精力做检索质量优化。

## 5. Phase 2：检索质量提升

### 5.1 阶段目标

在已有可用系统的基础上，提升“检索得到正确上下文”的概率和质量。

### 5.2 本期解决的问题

典型问题包括：

- 文档里明明有答案，但没有被召回
- 检索到了相关内容，但不够准确
- 多段信息综合问题命中上下文不全
- 召回结果噪音较高，影响回答可信度

### 5.3 本期交付能力

- 支持 dense + sparse 的混合召回思路
- 支持 rerank
- 支持 top_k 等检索参数调优
- 支持 baseline 与增强方案的对比验证

### 5.4 本期不做

- 复杂文档解析重构
- 多索引系统管理
- 大规模生产级观测体系
- 全自动训练与调参平台

### 5.5 完成标准

在同一组问题集上，系统比 Phase 1 更少出现“文档里有但没找到”的问题，且回答质量明显改善。

### 5.6 进入下一期条件

当检索策略已经初步稳定后，下一步主要瓶颈通常会转移到“文档本身解析得好不好”。

## 6. Phase 3：文档解析与进入系统能力

### 6.1 阶段目标

提升复杂 PDF、结构化文档和脏数据的处理能力，让系统拥有更好的知识原料。

### 6.2 本期解决的问题

- PDF 抽取顺序混乱
- 表格、标题、页眉页脚干扰 chunk
- 文档结构丢失导致检索不准
- 用户无法理解某些文档为什么效果差

### 6.3 本期交付能力

- 针对不同文件类型采用更合理的解析策略
- 保留 page、heading、section、source 等 metadata
- 结构化 chunking 能力初步建立
- 对复杂文档提供可诊断的处理链路

### 6.4 本期不做

- 完整多模态文档理解
- 企业级 OCR 平台能力
- 所有复杂格式一次性全支持

### 6.5 完成标准

对于复杂文档，系统解析结果明显优于 Phase 1，且回答引用更容易理解、检索命中更稳定。

### 6.6 进入下一期条件

当输入质量得到改善后，系统会进入“日常怎么更方便地使用和维护”的阶段。

## 7. Phase 4：可用性与运营能力

### 7.1 阶段目标

把系统从“能用一次”推进到“可以反复使用”。

### 7.2 本期解决的问题

- 文档更新后很难重新管理索引
- 不清楚哪些文档已经处理
- 不清楚错误发生在哪里
- 参数、状态、流程过于分散

### 7.3 本期交付能力

- 基础 index / corpus 管理能力
- 文档增删改后的重新索引流程
- 基础配置项梳理
- 状态提示与日志输出
- 常见故障定位信息

### 7.4 本期不做

- 团队级后台系统
- 复杂权限与角色管理
- 完整云端控制台

### 7.5 完成标准

用户不需要频繁改代码，就能重复完成“导入、索引、提问、更新文档、重新索引”的主要流程。

### 7.6 进入下一期条件

当系统已经进入可反复使用阶段，就必须建立评测和回归机制，否则每次优化都可能破坏已有能力。

## 8. Phase 5：评测与稳态建设

### 8.1 阶段目标

让系统迭代时有标准、有对比、有回归保护，不再依赖主观感觉。

### 8.2 本期解决的问题

- 每次改动后不知道是否真的更好
- 回答质量可能悄悄退化
- 没有统一评测问题集和检查流程
- 功能完成和质量稳定之间没有清晰边界

### 8.3 本期交付能力

- 标准问题集定义
- 评测结果记录方式
- baseline 对比流程
- groundedness / citation 检查机制
- 发布前检查清单

### 8.4 本期不做

- 全自动企业级评测平台
- 非常复杂的在线实验系统
- 大规模标注运营体系

### 8.5 完成标准

系统每次重要改动后，都可以按固定方式评测，并能明确判断是否存在质量回退。

## 9. 阶段依赖关系

整体依赖关系如下：

- Phase 1 是起点，没有它后续都没有基础
- Phase 2 依赖 Phase 1 的可用主链路
- Phase 3 依赖 Phase 1 / 2 暴露出来的真实输入问题
- Phase 4 依赖前几期已经形成了稳定可运行流程
- Phase 5 依赖前几期已有能力值得被持续保护和比较

## 10. 为什么这样分期是合理的

这套分期不是按技术组件罗列，而是按产品成长逻辑设计：

- 第一阶段确认产品有没有最小价值
- 第二阶段确认价值能不能变得更强
- 第三阶段确认上游输入能不能支撑更高质量
- 第四阶段确认产品能不能每天用
- 第五阶段确认产品能不能长期迭代

这种顺序的好处是，每一次投入都能换来新的真实能力，而不是陷入前期准备过重但无法交付的状态。

## 11. 每期统一交付要求

每一期在交付时，都必须明确回答以下问题：

1. 本期结束后，用户具体能做什么。
2. 本期与上一期相比新增了什么能力。
3. 本期明确不做什么。
4. 本期有哪些验收标准。
5. 本期如何通过手动或半自动方式验证。

只有满足这五点，这一阶段才算是真正可执行、可管理、可复用的需求阶段。
