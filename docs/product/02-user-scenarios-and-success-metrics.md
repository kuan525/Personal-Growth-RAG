# 用户场景与成功指标

## 1. 文档目的

本文件用于定义系统需要服务的真实使用场景，并为每个阶段建立能够手工验证、逐步量化的成功指标。

这个文档的重点不是学术评测，而是帮助产品建设过程中始终围绕真实使用价值前进。

## 2. 核心使用场景

## 场景 1：单文档事实问答

### 场景描述

用户上传一份 PDF 或 Markdown 文档，提出一个文档中明确出现过的事实性问题。

### 典型问题

- 这个项目支持哪些文件格式？
- 文档中介绍了哪些核心模块？
- 某个 README 里写的安装步骤是什么？

### 期望结果

- 系统能命中包含答案的 chunk
- 回答内容基本准确
- 能给出引用片段或来源

### 重要性

这是最基础、最能体现系统是否可用的场景。

## 场景 2：多 chunk 信息综合

### 场景描述

用户的问题需要结合文档中多个片段才能完整回答。

### 典型问题

- 这个系统的核心流程有哪些步骤，它们之间怎么衔接？
- 某个项目同时支持哪些输入和输出能力？
- 文档中对某个概念的定义和使用场景分别是什么？

### 期望结果

- 系统能召回多个相关 chunk
- 回答不是只摘取单段，而是能做基本归纳
- 引用能覆盖回答依据的主要部分

### 重要性

这是从“查找”走向“真正问答”的关键场景。

## 场景 3：术语解释与概念定位

### 场景描述

用户对某个术语、模块或名词不熟，希望系统告诉自己它是什么、在哪出现、和什么有关。

### 典型问题

- rerank 在这个系统里起什么作用？
- hybrid retrieval 和向量检索的区别是什么？
- chunking 在当前项目里承担什么职责？

### 期望结果

- 系统能找到相关上下文
- 回答不仅有定义，还能贴近项目上下文
- 引用能够帮助用户回到原文继续阅读

### 重要性

这个场景能显著提升学习效率，是个人知识库产品的重要价值来源。

## 场景 4：带证据的回答

### 场景描述

用户不只想看答案，还想知道答案是从哪里来的。

### 典型问题

- 这个结论来自哪一页？
- 这个回答的依据是什么？
- 哪些原文片段支持这个判断？

### 期望结果

- 每个回答尽量附带引用
- 引用可读且能帮助用户回到源文档
- 当证据不足时，系统应明确说明不确定

### 重要性

这是 RAG 产品区别于普通聊天助手的核心能力之一。

## 场景 5：无依据时拒答或降级回答

### 场景描述

用户提出的问题在文档中没有足够依据，系统需要避免编造。

### 典型问题

- 文档没提到的实现细节
- 文档外的扩展推断
- 与当前资料无关的问题

### 期望结果

- 系统能说明当前资料不足
- 可以给出“未找到明确证据”的提示
- 不应自信地编造文档中不存在的信息

### 重要性

如果没有这个能力，系统会很快失去可信度。

## 场景 6：导入后立即提问

### 场景描述

用户希望导入一批资料后，能快速开始提问，而不是经历复杂操作。

### 期望结果

- 导入流程清晰
- 索引完成后能快速提问
- 用户能知道哪些资料已经可用

### 重要性

这是产品使用门槛的关键。

## 场景 7：文档更新后继续可用

### 场景描述

用户增加、替换、删除资料后，希望系统仍然可维护。

### 期望结果

- 可识别哪些文档需要重新索引
- 更新后系统行为可预期
- 不会因为局部变更导致整体不可用

### 重要性

这是系统从 demo 走向工具的重要标志。

## 3. 成功指标设计原则

本系统的指标设计遵循以下原则：

- 优先可手工验证
- 优先真实使用价值，而非复杂学术指标
- 每一期指标都能在当前阶段落地
- 指标既覆盖结果，也覆盖流程体验

## 4. 通用成功指标

## 指标 1：文档导入成功率

### 含义

在当前支持的文件类型中，文档能顺利完成导入、抽取、切块和索引的比例。

### 为什么重要

如果连文档都无法稳定进入系统，后续问答质量没有讨论意义。

### 手工测量方式

准备一组测试文件，记录：

- 总导入数量
- 成功数量
- 失败数量
- 失败原因分类

## 指标 2：问题可回答率

### 含义

在测试问题集中，系统能给出可接受回答的比例。

### 为什么重要

这是最直接衡量系统有没有实际使用价值的指标。

### 手工测量方式

对问题集逐条判断：

- 回答正确
- 回答部分正确
- 回答错误
- 应拒答但未拒答
- 正确拒答

## 指标 3：引用覆盖率

### 含义

回答中附带引用或来源信息的比例。

### 为什么重要

这个系统的产品价值之一就是“答案有证据”。

### 手工测量方式

统计问题集中：

- 有引用回答数量
- 无引用回答数量
- 引用是否能回溯到原文

## 指标 4：已知答案命中率

### 含义

对于答案明确存在于文档中的问题，系统是否成功检索到包含核心信息的上下文。

### 为什么重要

可以区分问题是出在检索、解析还是回答阶段。

### 手工测量方式

给每道题预先标记参考文档和可能命中位置，测试时判断系统是否检索到正确来源。

## 指标 5：拒答正确率

### 含义

当问题缺乏文档证据时，系统是否能够合理拒答或说明不确定。

### 为什么重要

直接影响产品可信度。

### 手工测量方式

在问题集中加入若干无依据问题，统计：

- 正确拒答
- 错误编造
- 拒答过度

## 指标 6：重复使用便利度

### 含义

用户在导入、更新、提问、查看引用等流程中，完成一次完整使用的顺畅程度。

### 为什么重要

RAG 系统最终要变成工具，而不是一次性演示。

### 手工测量方式

记录典型流程中：

- 总操作步骤数
- 是否需要改代码
- 是否容易定位错误
- 是否容易知道当前状态

## 5. 各阶段重点指标

## Phase 1 重点指标

### 关注点

- 文档能否进入系统
- 简单问答是否可用
- 回答是否附带引用

### 建议重点指标

- 文档导入成功率
- 问题可回答率
- 引用覆盖率

### Phase 1 通过信号

- 用户可以拿自己资料进行基本问答
- 多数简单问题可用
- 回答通常能带证据

## Phase 2 重点指标

### 关注点

- 是否减少“明明有答案却没检索到”的问题
- 检索上下文质量是否提升

### 建议重点指标

- 已知答案命中率
- 问题可回答率
- 困难问题改善比例

### Phase 2 通过信号

- 同一问题集下，明显优于 Phase 1 baseline
- 检索命中更稳定

## Phase 3 重点指标

### 关注点

- 复杂文档能否更好地解析
- 结构化信息是否被保留
- 文档脏数据是否减少对检索的影响

### 建议重点指标

- 困难 PDF 导入成功率
- 复杂文档问答可回答率
- 引用可读性

### Phase 3 通过信号

- 复杂 PDF 不再明显拖垮系统质量
- 引用更容易理解和回溯

## Phase 4 重点指标

### 关注点

- 用户能否更轻松地重复使用系统
- 文档更新和索引维护是否更方便

### 建议重点指标

- 重复使用便利度
- 更新流程成功率
- 常见问题定位效率

### Phase 4 通过信号

- 用户不必频繁改代码就能维护资料库
- 主要操作流程可稳定重复

## Phase 5 重点指标

### 关注点

- 系统迭代时是否能稳定评估
- 是否能识别回归问题

### 建议重点指标

- 基线对比完成率
- 回归发现率
- 评测执行完整率

### Phase 5 通过信号

- 每次重要改动都能按同一方式评测
- 系统是否变好不再依赖主观感觉

## 6. 建议的问题集结构

为了后续验证，建议问题集至少覆盖以下类型：

- 事实型问题
- 多段综合问题
- 概念解释问题
- 容易混淆的问题
- 无依据问题
- 复杂文档问题

建议最初规模：

- Phase 1：15 ~ 20 题
- Phase 2：扩充到 25 ~ 40 题
- Phase 3 以后：加入困难 PDF 专项题

## 7. 成功的最终判断

如果产品建设到位，最终应体现为：

- 用户愿意长期把自己的资料导入系统
- 用户面对真实问题时愿意先问系统
- 系统给出的答案大多可被引用支撑
- 用户可以通过评测看见每期优化带来的改善
- 系统从“实验项目”成长为“个人知识工具”
