# 产品总览

## 1. 产品名称

本地文档问答助手

## 2. 产品定位

这是一个面向个人学习者、知识工作者和独立开发者的本地 RAG 系统，用来把分散的 PDF、Markdown、TXT 等资料整理成一个可以直接提问、可返回证据、可逐步迭代优化的文档问答助手。

它不是一个演示性质的玩具项目，也不是一套纯理论学习笔记，而是一个可以随着文档规模、检索质量和使用频率持续成长的本地知识系统。

## 3. 产品要解决的问题

当前用户在学习和使用 RAG 时，已经理解了不少概念，但缺少一个真正属于自己的、能够长期使用的系统。现实中的问题主要有：

- 资料很多，但分散在 PDF、Markdown、README 和各种学习文档中
- 看过一些内容，但需要时无法快速定位原文
- 只看概念时理解很多，真正落地时不知道先做什么
- 缺少一个可以不断验证、迭代、对比效果的工程对象
- 学习材料越来越多后，手工查找和人工归纳效率会明显下降

因此，这个产品要解决的核心问题不是“展示 RAG 可以做什么”，而是：

**把个人文档资料转化成一个可问、可答、可引用、可持续优化的本地知识问答系统。**

## 4. 目标用户

### 核心目标用户

- 正在学习 RAG 的个人开发者
- 希望通过实战而不是纯理论掌握 RAG 的使用者
- 有大量本地文档、项目 README、学习资料需要沉淀的人
- 时间有限，希望先做出可用系统，再逐步补知识的人

### 次级目标用户

- 需要对技术文档做快速问答和定位的人
- 想把个人知识库做成本地工具的人
- 想从 demo 逐步走向可用系统的 AI 应用开发者

## 5. 产品最终愿景

这个系统最终不是停留在“可以回答几个问题”的初级 demo，而是成长为一个具备以下特征的本地知识系统：

- 能稳定导入并管理多种格式文档
- 能针对问题检索出高质量上下文
- 能基于证据生成可信回答
- 能清晰展示引用来源和命中内容
- 能随着文档规模增长而持续维护
- 能通过评测和对比知道系统是否在变好
- 能让用户逐渐形成自己的 RAG 方法论和实现体系

一句话说，最终目标是：

**做一个自己愿意长期使用的本地 RAG 产品，而不是只跑通一次的实验脚本。**

## 6. 产品原则

### 6.1 本地优先

优先围绕本地运行、本地文件、本地使用场景进行设计，避免一开始引入复杂的远程基础设施和多人协作问题。

### 6.2 实用优先

先做出可用结果，再做结构升级。只要当前阶段能稳定解决真实问题，就优先交付，而不是为了完美架构推迟结果。

### 6.3 证据优先

系统给出的回答必须尽可能带引用、带来源、带原文片段。没有足够证据时，宁可明确说明不确定，也不追求看起来很流畅的编造答案。

### 6.4 渐进式建设

整个系统按阶段演进，从 MVP 到检索优化，再到解析增强、可用性建设、评测体系完善。每一阶段都必须能单独完成和验证。

### 6.5 可验证优先

每一阶段必须有清晰的验收标准。不能依赖“感觉做完了”或“看起来差不多”，而要能通过问题集、流程检查、手动测试或对比实验来验证。

## 7. MVP 承诺

第一阶段系统至少要做到：

- 支持导入 PDF / Markdown / TXT
- 支持文本抽取、切块、向量化和索引
- 支持用户提问
- 返回答案时附带原文引用或来源信息
- 对真实问题集具备基本可用性

MVP 的目标不是性能最强，也不是功能最全，而是让用户第一次真正拥有一个自己能用的 RAG 系统。

## 8. 为什么要分期建设

RAG 系统不是单点能力，而是一条完整链路：

- 文档进入系统
- 内容被解析和切块
- 索引和检索建立
- 回答生成
- 引用展示
- 后续迭代和评估

如果一开始试图把所有能力同时做好，会导致范围过大、验证困难、问题定位困难。因此本产品必须按阶段建设：

- 先确认系统能工作
- 再提升检索质量
- 再优化文档解析
- 再补运营和可维护能力
- 最后建立评测和稳态机制

这种分期方式的价值是：

- 每一步都能产出真实能力
- 每一步都能发现问题并沉淀认知
- 每一步都能为下一步提供明确输入

## 9. 当前明确不做的方向

为了保证节奏和可交付性，当前阶段不优先投入以下方向：

- 多用户协作与权限系统
- 云端同步和远程服务部署
- 企业级身份认证
- 多 Agent RAG 工作流
- GraphRAG
- 分布式向量数据库集群
- 大规模企业知识库治理
- 复杂多模态 OCR 全能力
- 大而全的工作流编排系统

## 10. 产品成功的标准

如果这个系统建设成功，应满足以下结果：

- 用户愿意拿自己的真实资料持续使用它
- 用户遇到文档问题时会优先选择提问系统，而不是手工翻资料
- 用户能清楚知道答案的依据来自哪里
- 用户能通过一次次迭代看见系统质量提升
- 用户能通过这个系统真正掌握 RAG 的工程实现路径
