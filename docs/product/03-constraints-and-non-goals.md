# 约束与非目标

## 1. 文档目的

本文件用于明确系统在当前阶段的边界条件、现实约束和暂不投入的方向，避免需求无限扩张，确保项目能够持续推进和真正交付。

## 2. 总体约束原则

本系统当前是一个面向个人使用的本地 RAG 产品，优先追求：

- 可落地
- 可验证
- 可持续迭代
- 与当前时间投入匹配

因此所有需求都必须服从以下判断标准：

1. 是否能让当前阶段更快形成可用结果。
2. 是否能被当前阶段独立验证。
3. 是否会显著增加复杂度但不带来立刻可见的产品价值。

## 3. 用户与使用场景约束

### 3.1 单用户优先

当前系统面向单一使用者，默认是个人本地知识库和个人文档问答场景。

当前阶段不考虑：

- 多用户协作
- 团队空间
- 权限隔离
- 成员级别的知识库管理

### 3.2 本地优先

当前系统默认运行在本地环境，面向个人文件目录和本地开发调试。

当前阶段不要求：

- 云端部署
- 远程服务托管
- SaaS 化多租户能力
- 在线管理控制台

### 3.3 文档问答优先

系统当前聚焦“文档问答”这个主场景，而不是通用 Agent 平台。

当前阶段不扩展到：

- 通用自动任务执行
- 复杂工具编排
- 跨系统工作流自动化

## 4. 数据范围约束

### 4.1 首批支持的文件类型

当前优先支持：

- PDF
- Markdown
- TXT

这是因为这三类文件足以覆盖大多数学习资料和 README 场景，也能支撑完整的 RAG 实战闭环。

### 4.2 暂缓支持的类型

暂时不优先支持：

- Word / PPT 等 Office 复杂格式
- 网页实时抓取
- 数据库直接接入
- 图片集直接理解
- 音视频资料

如果未来确实有使用场景，再单独纳入路线图。

## 5. 技术能力边界约束

### 5.1 先接受有限精度，不追求一步到位

在前几个阶段，系统允许存在：

- 对复杂 PDF 的解析不完美
- 对多段综合问题的回答还不够稳定
- 对无结构文档的效果有限
- 对召回结果的排序仍需后续优化

只要系统已经具备可用价值，就不应因为局部不完美而无限推迟交付。

### 5.2 先接受小规模数据集

系统前期默认处理的是小到中等规模的个人知识资料，而不是大规模企业级知识库。

因此前期不优先考虑：

- 分布式索引
- 超大 corpus 分片
- 高并发查询
- 横向扩展架构

### 5.3 先做可解释的链路，不追求过度自动化

当前系统应该更容易理解和调试，而不是追求过多自动智能行为。

因此前期优先：

- 明确的导入流程
- 明确的检索与回答过程
- 明确的引用展示
- 明确的错误定位

而不是优先投入：

- 黑盒式复杂推理流程
- 自动多策略决策引擎
- 难以解释的系统行为

## 6. 产品非目标

以下内容明确不属于当前产品主线：

## 6.1 非目标：多 Agent RAG 平台

当前目标是做好一个文档问答系统，不是搭建一个多 Agent 协作平台。

## 6.2 非目标：GraphRAG

GraphRAG 虽然有价值，但当前阶段不是优先级最高的问题。当前更重要的是基础链路、检索质量和输入质量。

## 6.3 非目标：企业级后台系统

当前不建设：

- 管理后台
- 组织级配置中心
- 复杂角色权限体系
- 企业治理与审计体系

## 6.4 非目标：复杂多模态系统

当前不把“图片理解、图表推理、版式视觉理解、OCR 全能力”作为第一优先目标，只在必要时逐步增强。

## 6.5 非目标：所有框架和向量库横评

当前项目不是技术选型研究报告，不追求一开始就比较大量框架与基础设施。

重点是：

- 先做出来
- 先用起来
- 先看问题出在哪

## 6.6 非目标：一开始就做完美架构

当前阶段不为了未来假设需求做过度抽象，不为了“可能以后需要”提前把复杂结构全搭好。

## 7. 资源与节奏约束

### 7.1 时间有限

当前产品建设必须承认现实：用户时间有限，优先选择短路径形成结果。

这意味着：

- 先做 MVP
- 再看问题补能力
- 每一期都要尽量可单独交付

### 7.2 学习方式以实践驱动为主

本产品路线默认用户更适合：

- 先动手
- 遇到问题再补知识
- 用项目逼出学习需求

因此产品文档必须服务于实现，不应写成过度理论化的架构教材。

## 8. 风险边界

以下风险需要在规划时持续意识到：

- 如果范围不加控制，项目会快速膨胀
- 如果没有明确验收标准，每一期都容易变成“做了很多但无法证明完成”
- 如果过早进入复杂架构设计，会牺牲实际交付速度
- 如果不限制非目标，系统可能偏离“本地文档问答助手”的核心价值

## 9. 如何使用这些约束

后续每新增一个需求，都建议先问四个问题：

1. 它是否直接提升当前阶段的产品价值？
2. 它是否属于当前阶段范围，而不是未来阶段？
3. 它是否会显著增加复杂度？
4. 它是否能在当前阶段被验证？

如果无法明确回答，就不应轻易纳入当前阶段。

## 10. 结论

这些约束不是为了保守，而是为了确保项目能够从概念走向真实产品。

真正合理的产品规划，不是把所有可能做的事情都写进去，而是知道：

- 现在最应该做什么
- 现在明确不做什么
- 为什么这些取舍是必要的

只有这样，系统才能稳定推进，并最终长成一个真正可用的本地 RAG 产品。
