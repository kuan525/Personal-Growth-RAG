# 系统架构设计

## 1. 架构目标

本系统架构服务的是长期个人知识系统，而不是一次性 RAG pipeline。

架构必须优先保证：

- answer 可追溯
- document 可 incremental update
- metadata 一致
- graph 和 memory 可长期增长
- evaluation 和 regression 可执行
- local-first

## 2. 高层组件

```text
Sources
  -> Ingestion
  -> Normalization
  -> Chunking
  -> Metadata Store
  -> Vector Index / Sparse Index
  -> Retrieval / Rerank
  -> Answering
  -> Evaluation

                         -> Entity Extraction
                         -> Graph Store
                         -> Memory Store
                         -> Decision Support
```

## 3. 组件职责

### 3.1 Source Manager

负责发现文件并分配稳定 document identity。

职责：

- 扫描 source directories
- 识别支持的 file types
- 计算 content hash
- 判断 new / modified / deleted / unchanged
- 维护 document status

### 3.2 Parser

负责从支持格式中抽取 raw content。

职责：

- 解析 TXT
- 解析 Markdown，并保留 heading structure
- 解析 PDF，尽可能保留 page metadata
- 清晰报告 parse errors

### 3.3 Normalizer

负责把 raw content 转成更干净的 structured blocks。

职责：

- 统一 whitespace
- 保留 paragraph boundary
- 保留 heading path
- 安全地去除明显噪音
- 生成 block-level intermediate representation

### 3.4 Chunker

负责把 blocks 转换为可 retrieval 的 chunks。

职责：

- 创建语义上相对完整的 chunk
- 保留 source metadata
- 分配 chunk ID
- 保留 chunk order
- 避免混合不相关 section

### 3.5 Metadata Store

负责保存系统事实状态。

初始推荐：SQLite。

存储对象：

- documents
- chunks
- query runs
- retrieval results
- answers
- citations
- entities
- relations
- memories
- decisions
- evaluation runs

### 3.6 Vector Index

负责 semantic retrieval。

MVP：

- FAISS

后续：

- 当 update/delete/filter 需求增强后，引入 Qdrant

### 3.7 Sparse Index

负责 lexical retrieval。

可选方案：

- BM25
- SQLite FTS
- local search library

### 3.8 Retriever

负责 candidate recall。

职责：

- dense retrieval
- sparse retrieval
- metadata filter
- hybrid merge
- retrieval_source tracking

### 3.9 Reranker

负责按 query relevance 重新排序 candidates。

职责：

- rerank merged candidates
- 保留 original rank 和 reranked rank
- 选择 final context

### 3.10 Answer Generator

负责 grounded answer generation。

职责：

- 组装 context
- 生成 answer
- 输出 citation
- evidence 不足时拒答或说明不确定
- 记录 answer trace

### 3.11 Graph Builder

负责抽取高价值 entities 和 relations。

职责：

- 抽取 project / skill / goal / decision / problem / resource
- 抽取有 evidence 的 relation
- 维护 entity mentions
- 当 evidence 变化时 invalidated graph entries

### 3.12 Memory Manager

负责长期 personal memory。

职责：

- 生成 memory candidates
- 要求 source evidence
- 区分 temporary fact 和 long-term fact
- 支持 correction 和 invalidation

### 3.13 Decision Engine

负责结构化 decision support。

职责：

- 分类 decision question
- 检索 relevant history
- 比较 options
- 区分 evidence 与 inference
- 记录 decision 和 review outcome

### 3.14 Evaluation Runner

负责 testing 和 regression。

职责：

- 运行 question set
- 捕获 answers 和 citations
- 记录人工 judgment
- 与 baseline 对比
- 分类 failures

## 4. 数据流

### 4.1 Ingestion Flow

1. 扫描 source path。
2. 计算 file metadata 和 hash。
3. 判断状态：new、modified、deleted、unchanged。
4. 解析变化的 source。
5. Normalize 成 blocks。
6. Chunk blocks。
7. 存储 documents 和 chunks。
8. 生成 embeddings。
9. 更新 vector index 和 sparse index。
10. 可选触发 graph extraction。

### 4.2 Query Flow

1. 接收 query。
2. 分类 query type。
3. 应用 optional filters。
4. 执行 dense retrieval。
5. 执行 sparse retrieval。
6. Merge candidates。
7. Rerank candidates。
8. 可选 graph expansion。
9. 组装 final context。
10. 生成 answer 和 citations。
11. 存储 query run 和 answer trace。

### 4.3 Decision Flow

1. 接收 decision question。
2. 识别 decision object、constraints、time range。
3. 检索相关 documents、memories、decisions、graph relations。
4. context 不足时提出 clarification。
5. 比较 options。
6. 生成 recommendation、evidence 和 risk。
7. 保存 decision record 以供后续 review。

## 5. 分阶段架构演进

### Phase 1

- Python CLI
- SQLite metadata
- FAISS vector index
- basic parser
- basic chunking
- answer with citations

### Phase 2

- sparse index
- hybrid retrieval
- rerank
- evaluation runner

### Phase 3

- structured blocks
- better metadata
- parser diagnostics

### Phase 4

- incremental update engine
- document / chunk status
- index invalidation

### Phase 5

- entities
- relations
- entity mentions
- graph-assisted retrieval

### Phase 6

- review ingestion
- memory candidates
- time-based summaries

### Phase 7

- decision records
- decision support prompts
- decision review loop

## 6. 架构规则

1. 没有 retrieval trace，不生成最终 answer。
2. 没有 evidence，不创建 active graph relation。
3. deleted document 不能作为 active retrieval source。
4. 没有 source，不创建 confirmed memory。
5. 没有 evaluation evidence，不接受所谓质量提升。
6. 早期不要让 UI 反向驱动核心架构。
