# 检索与回答链路规格说明

## 1. 文档目的

本文件定义系统从用户提问到返回答案的核心链路规范，包括 query 输入、召回流程、上下文组装、回答生成、引用格式与拒答策略。

它的目标是为系统提供一个跨阶段稳定的主链路契约。

## 2. 适用范围

本规格适用于：

- 用户提问处理
- 检索链路
- 上下文选择
- 回答生成
- 引用展示
- 无依据问题的处理策略

## 3. 标准链路总览

建议将问答主链路定义为以下阶段：

1. 接收 query
2. query 预处理
3. 初始召回
4. 候选上下文排序
5. 上下文组装
6. 回答生成
7. 引用输出
8. 结果记录

## 4. query 输入规范

每次提问至少包含：

- query_text
- request_time
- session_id（如果有）
- 当前 corpus / collection 信息（如果有）

### 可选附加信息

- 用户指定文档范围
- 用户指定问题类型
- 用户设定的 top_k
- 用户是否强制要求引用

## 5. query 预处理规范

### 当前最低要求

- 接收自然语言问题
- 基础清洗输入
- 保持原始 query 可追溯

### 后续可增强方向

- query rewrite
- query classification
- query expansion

除非进入后续阶段，否则预处理不应过重，以免增加不必要复杂度。

## 6. 初始召回规范

### Phase 1 基线

- 基于向量检索进行初始召回
- 返回 top_k 候选 chunk

### Phase 2 增强

- 增加 sparse / BM25 路径
- 支持 dense + sparse 混合召回
- 记录召回来源

### 输出要求

初始召回结果至少应包含：

- chunk_id
- document_id
- chunk_text
- score
- retrieval_source
- 基础 metadata

## 7. 候选排序规范

### 基础模式

- 按初始 score 排序

### 增强模式

- 支持 rerank
- rerank 后需保留排序前后信息

### 记录要求

排序结果建议记录：

- original_rank
- reranked_rank
- final_selected

## 8. 上下文组装规范

### 要求

- 只将最终入选的候选 chunk 送入回答阶段
- 上下文数量应可配置
- 尽量避免大量噪音 chunk 混入
- 需要保留 chunk 与来源信息的映射关系

### 不建议情况

- 所有召回内容直接全部塞给模型
- 引用与实际输入模型的上下文不一致

## 9. 回答生成规范

### 核心要求

- 回答必须尽量基于检索上下文
- 不能把无依据推测伪装成文档事实
- 当证据不足时，应允许保守回答或拒答

### 推荐回答原则

1. 先回答用户问题
2. 再给出主要依据
3. 对不确定部分明确说明边界
4. 尽量避免无证据扩展

## 10. 引用格式规范

回答应尽量附带引用，引用至少包含：

- 来源文件名
- 原文片段

若可获得，建议增加：

- 页码
- 标题 / 章节
- chunk 顺序信息

### 引用输出要求

- 引用与回答使用的实际上下文保持一致
- 引用应能帮助用户回到源文档
- 引用不应只有抽象编号而没有可读内容

## 11. 拒答与不确定回答规范

### 适用场景

- 未检索到足够证据
- 检索结果与问题明显不匹配
- 证据冲突且当前无法判断

### 处理原则

- 明确告知当前资料中未找到充分依据
- 可以提示用户换种问法或检查文档范围
- 不应在没有支持证据时做高置信断言

### 通过标准

一个合格的 RAG 系统应具备“合理拒答”的能力，而不是一味输出流畅文本。

## 12. 结果记录规范

每次问答建议记录以下信息：

- query_text
- selected_chunks
- retrieval_source
- final_answer
- citations
- whether_refused
- timestamp
- config_snapshot（如果有）

这些记录将用于评测、调试和对比分析。

## 13. 质量判断维度

每次结果至少可从以下维度判断：

- 是否命中正确来源
- 回答是否基于证据
- 回答是否完整
- 引用是否可用
- 无依据时是否正确拒答

## 14. 验收建议

本规格对应的主链路能力至少应满足：

- 用户问题可被系统处理
- 系统能检索出候选上下文
- 系统能返回回答和引用
- 引用与回答所依据的上下文一致
- 证据不足时不会频繁编造

## 15. 与阶段路线的关系

- Phase 1：建立最小可用主链路
- Phase 2：增强召回与排序质量
- Phase 5：基于本规格建立回归与评测规则

因此，本规格是整个系统最核心的跨阶段规范之一。
