# 文档进入系统规格说明

## 1. 文档目的

本文件定义文档进入系统时的统一规范，包括支持类型、抽取规则、规范化策略、切块前中间表示、重复处理方式和典型错误分类。

它的作用是让不同阶段的导入和解析能力共享同一套基础约束，避免每一阶段都重新定义“文档怎么进入系统”。

## 2. 适用范围

本规格适用于：

- 文档导入
- 文本抽取
- 文档规范化
- chunk 前处理中间表示
- 文档更新与重复识别
- 导入错误分类

## 3. 支持文件类型

### 当前优先支持

- PDF
- Markdown
- TXT

### 后续可能扩展

- DOCX
- HTML
- 图片 OCR 输入
- 更多结构化文档格式

除非路线图明确进入新阶段，否则当前只承诺前三类。

## 4. 导入流程总览

标准导入流程建议包括以下步骤：

1. 文件发现
2. 文件类型识别
3. 原始内容抽取
4. 规范化处理
5. 中间表示生成
6. chunk 生成
7. metadata 绑定
8. 进入 embedding / index 流程

## 5. 文件发现规则

### 要求

- 系统应明确知道本次处理哪些文件
- 文件路径必须可追溯
- 不支持类型应被识别并跳过或提示

### 建议保留信息

- source_path
- source_name
- file_extension
- file_size
- last_modified_at

## 6. 文本抽取规则

## 6.1 TXT

### 规则

- 按纯文本读取
- 保留原始顺序
- 尽量不做多余结构推断

## 6.2 Markdown

### 规则

- 保留标题层级信息
- 保留列表、代码块、段落边界
- 尽量保留基础结构，而不是直接压扁成大段纯文本

## 6.3 PDF

### 规则

- 至少获取可读文本
- 若可行，保留页码信息
- 对复杂 PDF 可走增强解析路径
- 需要尽量减少页眉页脚噪音对正文的污染

## 7. 文档规范化规则

抽取后的文本不应直接进入 chunking，建议先做轻量规范化。

### 建议规范化内容

- 统一换行与空白字符
- 去除明显空块
- 标记或清理重复性页眉页脚
- 维护段落顺序
- 保留标题和段落边界

### 不建议过度处理

- 不要过早丢失原始结构
- 不要把所有文本合成单一字符串而失去来源边界

## 8. 中间表示规范

在抽取与切块之间，建议形成统一中间表示。每个文档可被表示为若干结构块，结构块至少包含：

- block_id
- block_type
- text
- source_path
- page_number（如果有）
- heading_path（如果有）
- sequence_order

### block_type 建议值

- title
- heading
- paragraph
- list_item
- table_text
- code_block
- unknown

这个中间表示是后续结构化 chunking 和调试的基础。

## 9. chunk 生成规则

### 基本要求

- chunk 应尽量具备语义完整性
- chunk 需要可追溯到来源文档和来源位置
- chunk 不应跨越明显不相关的结构边界

### 建议保留字段

- chunk_id
- document_id
- chunk_text
- source_path
- chunk_order
- page_start / page_end（如果有）
- heading_path（如果有）
- parser_type

## 10. 重复文档与更新文档处理

### 目标

避免用户反复导入后系统中出现无法区分的新旧内容。

### 建议处理策略

- 对每个文档分配稳定 document_id
- 记录文件路径和最后修改时间
- 对更新文档支持重新处理与重新索引
- 对删除文档支持移除对应索引内容

### 不建议状态

- 同一路径文件更新后形成多份无法区分的旧数据
- 删除原文档后索引仍长期保留旧内容且无提示

## 11. 典型错误分类

导入问题建议至少分类为以下类型：

### 11.1 文件不可读

- 路径错误
- 权限问题
- 文件损坏

### 11.2 类型不支持

- 当前系统未支持的扩展名

### 11.3 抽取失败

- PDF 文本无法正常抽取
- 编码问题导致 TXT / Markdown 无法读取

### 11.4 解析质量低

- 输出文本顺序混乱
- 大量噪音污染
- 结构信息严重缺失

### 11.5 chunk 生成异常

- chunk 为空
- chunk 过大或过小
- chunk 数量异常

## 12. 验收建议

本规格对应的导入能力应至少满足：

- PDF / Markdown / TXT 三类文件可进入统一流程
- 文档来源信息可追溯
- chunk 能被后续检索和引用使用
- 错误具备基础分类，便于定位

## 13. 与后续阶段关系

- Phase 1 依赖本规格建立最小导入闭环
- Phase 3 依赖本规格增强复杂文档解析
- Phase 4 依赖本规格管理文档更新和删除

因此该规格应保持稳定，并在后续演进中补充，而不是频繁推翻重来。
