# 评测规格说明

## 1. 文档目的

本文件定义系统的评测方式，包括问题集格式、问题分类、人工审核标准、结果记录方式与回归比较原则。

其目标不是建立复杂学术评测体系，而是建立一套适合当前项目阶段、可持续执行的产品质量评估方法。

## 2. 评测设计原则

评测体系应满足以下原则：

- 真实：问题尽量来自真实使用场景
- 轻量：适合个人项目长期执行
- 可重复：不同版本可进行对比
- 可解释：结果能定位到具体问题类别
- 可行动：评测结果能指导下一步优化

## 3. 问题集结构

每个问题建议至少包含以下字段：

- question_id
- question_text
- question_type
- source_scope
- expected_behavior
- reference_source（如果有）
- notes（可选）

## 4. 问题分类建议

建议至少包含以下类型：

### 4.1 事实型问题

答案明确存在于某一段或某一页中。

### 4.2 多段综合问题

需要结合多个 chunk 才能较完整回答。

### 4.3 概念解释问题

需要解释术语、模块含义或上下文角色。

### 4.4 困难检索问题

文档中有答案，但关键词不明显、表述分散或易混淆。

### 4.5 无依据问题

文档中没有足够依据，系统应拒答或保守回答。

### 4.6 复杂文档问题

来自复杂 PDF 或结构不佳文档，用于验证解析与检索增强效果。

## 5. 评测集规模建议

### Phase 1

- 15 ~ 20 题

### Phase 2

- 25 ~ 40 题

### Phase 3 以后

- 增加复杂文档专项题
- 增加困难检索题

## 6. 结果记录格式

每次评测建议记录：

- question_id
- 当前版本
- 当前配置摘要
- 检索到的主要来源
- 最终回答
- 最终引用
- 人工评估结果
- 备注

## 7. 人工评估维度

每个问题至少从以下维度评估：

### 7.1 是否命中正确来源

- 是
- 部分命中
- 否

### 7.2 回答是否基于证据

- 明显基于证据
- 部分基于证据
- 脱离证据

### 7.3 回答是否足够完整

- 完整
- 部分完整
- 明显缺失

### 7.4 引用是否可用

- 可用
- 有引用但帮助有限
- 无引用或引用错误

### 7.5 无依据时是否正确拒答

- 正确拒答
- 应拒答但未拒答
- 不该拒答却拒答

## 8. 总体判断方式

每道题最终可给出一个综合结论：

- 通过
- 部分通过
- 失败

同时建议记录失败原因属于哪一类：

- 解析问题
- 检索问题
- rerank 问题
- 回答问题
- 引用问题
- 拒答问题

## 9. baseline 对比原则

每次重要优化后，都建议与一个稳定 baseline 对比。对比内容至少包括：

- 通过题数变化
- 失败类别变化
- 引用质量变化
- 困难问题改善情况
- 无依据问题处理情况

## 10. 回归检查集

建议从完整问题集中抽取一组核心题，作为回归检查集。其作用是：

- 覆盖系统关键能力
- 每次重要改动后快速执行
- 尽早发现明显退化

回归检查集建议覆盖：

- 事实问题
- 困难检索问题
- 无依据问题
- 复杂文档问题

## 11. 输出报告建议

每次评测结束后，建议至少输出：

- 本次测试版本
- 问题总数
- 通过 / 部分通过 / 失败数量
- 主要提升点
- 主要退化点
- 下一步优化建议

## 12. 验收建议

一个可用的评测体系至少应做到：

- 同一问题集可重复执行
- 不同版本结果可比较
- 能区分问题类型和失败原因
- 能对产品迭代提供明确反馈

## 13. 与产品路线的关系

- Phase 1：建立初始问题集
- Phase 2：用于比较检索增强效果
- Phase 3：用于比较复杂文档解析效果
- Phase 5：正式沉淀为稳态机制

因此，评测规格既服务于开发，也服务于产品决策。
