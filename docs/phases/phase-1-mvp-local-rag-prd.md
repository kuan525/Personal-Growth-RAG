# Phase 1 PRD：MVP 本地 RAG 基线

## 1. 阶段目标

本阶段的目标是交付一个**真正可用的本地文档问答 MVP**。它不是展示概念的样例，而是一个能够导入真实资料、接受真实提问、返回带证据回答的最小产品。

一句话目标：

**让用户第一次拥有一个可以直接拿自己资料来问的本地 RAG 系统。**

## 2. 为什么现在先做这一期

在当前阶段，最大的风险不是“效果还不够极致”，而是“始终停留在理解和准备阶段，没有自己的可运行系统”。

因此第一期必须解决最核心的问题：

- 文档能不能进入系统
- 系统能不能建立索引
- 用户能不能提问
- 回答能不能附带证据

只有当这条最小链路成立，后续关于 hybrid retrieval、rerank、复杂 PDF 解析、评测体系等建设才有落点。

## 3. 用户问题

本期主要解决以下问题：

- 本地资料太多，难以快速找到原文
- 已经了解 RAG 概念，但没有真正属于自己的系统
- 想通过实战掌握完整流程，但不知道从哪里下手
- 需要一个可用的起点，而不是继续堆理论

## 4. 本期范围

本期必须覆盖以下能力：

1. 文档导入
2. 文本抽取
3. 文本切块
4. embedding 生成
5. 本地向量索引建立
6. 用户提问
7. 检索相关 chunk
8. 生成回答
9. 返回引用或来源信息
10. 用真实问题集做基础验证

## 5. 本期不做

为了保证 MVP 可完成，本期明确不做：

- hybrid retrieval
- rerank
- query rewrite
- 高级 PDF layout 解析
- OCR 强化链路
- 文档变更自动监听
- 复杂 corpus 管理
- 多用户与权限
- 完整自动化评测系统
- 复杂 Web 控制台

## 6. 目标用户与使用方式

### 目标用户

- 个人学习者
- 独立开发者
- 想把本地学习资料做成问答系统的人

### 典型使用方式

1. 准备若干本地文档
2. 导入系统
3. 建立索引
4. 提出真实问题
5. 查看回答与引用
6. 根据效果判断下一步优化方向

## 7. 用户流程

### 流程 1：首次导入与建立索引

1. 用户准备一组本地 PDF / Markdown / TXT 文件
2. 系统读取文件
3. 系统抽取文本内容
4. 系统对文本进行切块
5. 系统生成 embedding
6. 系统建立本地索引
7. 系统提示索引完成

### 流程 2：提问并获取回答

1. 用户输入问题
2. 系统将问题向量化或转换为检索输入
3. 系统从索引中召回相关 chunk
4. 系统把上下文交给 LLM 生成回答
5. 系统返回回答与来源片段

### 流程 3：问题无依据时的降级处理

1. 用户提出超出文档范围的问题
2. 系统检索不到足够证据
3. 系统提示未找到明确依据或回答不确定
4. 系统避免输出看似流畅但缺乏证据的结论

## 8. 功能需求

## 8.1 文档导入

### 要求

- 支持导入 PDF、Markdown、TXT
- 支持一次导入一个或多个文档
- 能识别文件路径和文件类型
- 对不支持文件类型给出明确提示

### 验收点

- 三种支持类型都能进入处理流程
- 非支持类型会被明确拒绝或跳过

## 8.2 文本抽取

### 要求

- 能从支持文件中抽取可用于后续切块的文本
- 保留基础来源信息
- PDF 至少能得到可阅读文本

### 验收点

- 抽取文本非空
- 用户能看到来源文件信息
- 失败时有错误提示

## 8.3 文本切块

### 要求

- 使用一个简单、可解释的 chunking 方案
- chunk 大小与 overlap 有默认值
- 每个 chunk 应能关联来源文档

### 验收点

- 文本不会全部塞进一个 chunk
- 切块结果可被检索与引用
- chunk 数量和结果基本符合预期

## 8.4 embedding 与索引

### 要求

- 为 chunk 生成 embedding
- 建立本地索引
- 小规模文档下索引构建可完成

### 验收点

- 索引构建流程可正常结束
- 后续可进行检索
- 索引失败时可定位问题

## 8.5 提问与检索

### 要求

- 系统接收自然语言问题
- 返回一组相关 chunk 作为上下文
- 支持配置基础 top_k

### 验收点

- 提问后能返回候选上下文
- 简单问题能命中包含答案的 chunk

## 8.6 回答生成

### 要求

- 系统基于检索上下文回答
- 回答尽量限制在已检索证据范围内
- 证据不足时可以选择保守回答或拒答

### 验收点

- 回答不应明显脱离文档上下文
- 对无依据问题不能频繁编造

## 8.7 引用展示

### 要求

- 回答中或回答后展示来源信息
- 至少能展示命中的原文片段和来源文件名
- 若有页码信息，可一并展示

### 验收点

- 大多数回答都附带引用
- 用户能根据引用回到文档继续查看

## 9. 非功能需求

## 9.1 可理解性

整个 MVP 应足够简单，方便后续调试和扩展。用户应大致知道文档如何进入系统、如何被检索、如何生成回答。

## 9.2 可运行性

在本地环境下，小规模文档集能稳定运行，不要求高并发和高吞吐。

## 9.3 可定位问题

出现失败时，至少能判断问题出在：

- 导入阶段
- 抽取阶段
- 切块阶段
- 索引阶段
- 检索阶段
- 回答阶段

## 10. 数据与元数据要求

每个文档至少应具备：

- document_id
- 文件路径或文件名
- 文件类型

每个 chunk 至少应具备：

- chunk_id
- 所属 document_id
- chunk 文本
- 基础顺序信息
- 引用所需来源信息

若能获取，建议保留：

- 页码
- 标题层级
- section 信息

## 11. 风险与取舍

## 风险 1：PDF 抽取质量有限

### 说明

Phase 1 接受 PDF 抽取并不完美，但不能因此阻塞整个 MVP。

### 取舍

优先先让简单 PDF 可用，复杂 PDF 在后续阶段解决。

## 风险 2：回答可能看起来通顺但证据不充分

### 说明

这是典型 RAG 风险。

### 取舍

优先要求引用和保守回答策略，宁可少答也不要过度编造。

## 风险 3：前期不做复杂优化导致效果一般

### 说明

这是预期内结果。

### 取舍

Phase 1 重点不是最佳效果，而是形成可用闭环。

## 12. 依赖项

本期依赖以下条件：

- 有一组可供测试的本地文档
- 有一个可用的 embedding 与 LLM 方案
- 有一个本地向量索引方案
- 有可运行的开发环境

## 13. 验收标准

以下条件全部满足，视为本期完成：

1. 至少支持 PDF、Markdown、TXT 三类文件进入系统
2. 至少能完成一次从导入到提问的完整链路
3. 至少能基于 15 ~ 20 个真实问题进行测试
4. 多数简单问题回答基本可用
5. 回答通常附带来源片段或引用信息
6. 对超出资料范围的问题，不应频繁编造

## 14. 验证方法

## 14.1 测试资料准备

准备至少：

- 1 份 PDF
- 1 份 Markdown
- 1 份 TXT

内容尽量来自真实资料，而非专门为了演示构造的极简样例。

## 14.2 问题集准备

建议准备 15 ~ 20 个问题，覆盖：

- 文档中明确写过的事实问题
- 术语解释问题
- 需要跨两段理解的问题
- 无依据问题

## 14.3 验证步骤

1. 导入文档
2. 建立索引
3. 对问题集逐条提问
4. 记录回答情况
5. 检查是否带引用
6. 对错误案例分类

## 14.4 验证输出

建议至少记录：

- 问题
- 回答
- 是否命中正确来源
- 是否带引用
- 是否需要拒答
- 失败原因

## 15. 本期结束后的产品状态

Phase 1 完成后，产品应达到：

- 不是概念模型，而是一个真实可用的本地问答工具
- 用户已经可以把自己的文档变成可提问对象
- 系统已具备后续优化的稳定基线

## 16. 进入 Phase 2 的条件

当以下情况成立时，进入 Phase 2：

- MVP 已经可跑通
- 用户已经开始用真实资料提问
- 错误主要集中在“检索质量不够好”而不是“系统根本跑不通”
