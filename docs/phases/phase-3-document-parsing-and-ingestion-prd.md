# Phase 3 PRD：文档解析与进入系统能力

## 1. 阶段目标

本阶段的目标是提升文档进入系统时的质量，尤其是复杂 PDF、结构化内容和脏数据文档的处理效果，让后续检索和回答建立在更好的原材料之上。

一句话目标：

**让系统不只是会问答，还要尽可能把文档“读对、切对、保留对”。**

## 2. 为什么现在做这一期

当 Phase 1 与 Phase 2 完成后，系统已经具备基本问答能力与一定的检索质量优化能力。此时如果仍然经常出现效果差，问题往往来自更上游：

- 文档抽取顺序错乱
- 表格和正文混在一起
- 标题层级丢失
- 页眉页脚污染内容
- chunk 边界语义质量差

如果上游输入质量差，后面再优化检索与回答，收益都会受限。因此这一阶段的重点，是提升文档被系统“吸收”的质量。

## 3. 用户问题

本期主要解决以下问题：

- 某些 PDF 抽取后文本阅读顺序混乱
- 表格信息丢失或碎裂
- 标题、章节、正文的结构关系丢失
- 页码、段落边界等上下文信息缺失
- 用户知道某文档效果差，但不知道问题出在哪里

## 4. 本期范围

本期必须覆盖以下能力：

1. 区分不同文件类型的进入策略
2. 提升复杂 PDF 的解析能力
3. 引入结构化或半结构化中间表示
4. 增强 metadata 保留
5. 优化 chunking 对文档结构的利用
6. 为失败文档提供更可诊断的处理信息

## 5. 本期不做

为了控制范围，本期不做：

- 全量多模态文档理解
- 所有图片、图表的高级推理
- 企业级 OCR 平台建设
- 完整通用文档 ETL 平台
- 所有格式一次性全支持

## 6. 用户流程

### 流程 1：导入结构较简单的文档

1. 用户导入 Markdown / TXT / 简单 PDF
2. 系统选择基础解析路径
3. 系统保留基础来源和结构信息
4. 系统进行更合理的 chunking

### 流程 2：导入复杂 PDF

1. 用户导入复杂 PDF
2. 系统识别需要增强解析路径
3. 系统提取文本、结构、页码、标题等信息
4. 系统对异常情况进行标记
5. 系统生成更可用于检索的 chunk

### 流程 3：诊断低质量文档

1. 用户发现某份文档问答效果差
2. 系统可回看解析结果或中间状态
3. 用户判断问题来自解析、结构缺失还是原文质量
4. 决定是否采用更强前处理方式

## 7. 功能需求

## 7.1 文件类型分流处理

### 要求

- 系统根据文件类型选择合适的解析策略
- 简单文本文件保持轻量链路
- 复杂 PDF 可以进入增强链路

### 验收点

- 不同类型文档不会全部走同一粗糙路径
- 用户能知道某文档采用了哪类处理方式

## 7.2 增强 PDF 解析

### 要求

- 改善文本阅读顺序
- 尽量保留页面信息
- 对标题、段落、正文、表格建立更清晰边界
- 减少页眉页脚等噪音影响

### 验收点

- 复杂 PDF 的文本可读性明显优于基础抽取
- 引用结果更容易让用户理解来源

## 7.3 中间表示规范

### 要求

- 在抽取与切块之间引入更稳定的中间文档表示
- 中间表示应承载文本、结构和来源信息
- 后续 chunking 依赖该表示进行处理

### 验收点

- chunking 不再只基于原始纯文本
- 结构信息可用于后续检索和引用

## 7.4 metadata 增强

### 要求

在本期基础上尽量保留：

- source
- page
- heading
- section
- 文档顺序信息
- 解析路径信息

### 验收点

- 回答引用更清晰
- 后续过滤与调试基础更完整

## 7.5 结构化 chunking

### 要求

- chunk 边界尽量与语义和结构一致
- 尽量避免把不同章节或不同语义块粗暴拼接
- 对表格、列表、标题附近内容做特殊处理

### 验收点

- chunk 语义完整度优于早期纯字符切块
- 检索命中更准确

## 7.6 诊断信息输出

### 要求

- 对解析失败或低质量结果给出可诊断信息
- 用户可以知道问题大致来自哪一层

### 验收点

- 用户不再只能看到“效果不好”，而能知道“为什么不好”

## 8. 非功能需求

## 8.1 可扩展性

本期的解析链路应允许后续继续增强，而不是只能适配一种特定文档。

## 8.2 可理解性

即使引入更复杂前处理，也应尽量让用户理解各阶段作用，避免整个入口链路变成完全黑盒。

## 8.3 容错性

对于解析质量不佳的文档，系统应允许降级使用，而不是一旦碰到复杂文档就整体失败。

## 9. 数据与元数据要求

本期需要进一步标准化以下信息：

- document_id
- parser_type
- source_path
- page_number
- heading_path
- section_label
- chunk_order
- raw_text / normalized_text 关系
- 解析异常标签

这些字段将用于后续检索、引用、过滤和问题定位。

## 10. 风险与取舍

## 风险 1：解析能力增强后实现复杂度明显上升

### 说明

相比前两期，本期更接近数据工程和前处理系统建设。

### 取舍

优先处理最常见、收益最大的复杂文档问题，不追求一步覆盖所有格式与版式。

## 风险 2：解析更复杂但收益难以感知

### 说明

如果没有困难文档测试集，增强效果可能不明显。

### 取舍

必须准备复杂 PDF 专项样本，且验证方式要能对比前后差异。

## 风险 3：链路过长影响维护

### 说明

文档进入系统后阶段越多，出错点也越多。

### 取舍

尽量保持中间表示清晰、日志明确、分层边界稳定。

## 11. 依赖项

本期依赖：

- Phase 1 已有导入和问答主链路
- Phase 2 已能初步区分检索问题与输入问题
- 已有若干复杂 PDF 作为困难测试样本

## 12. 验收标准

满足以下条件可视为本期完成：

1. 至少对复杂 PDF 提供更优于基础方案的解析路径
2. 引用中可展示更多可理解的来源信息
3. metadata 明显比前几期更完整
4. 对困难文档的问答效果优于旧链路
5. 解析失败或低质量时，能提供可诊断信息

## 13. 验证方法

## 13.1 测试资料准备

准备两类文档：

- 简单文档：格式清晰、结构简单
- 困难文档：复杂 PDF、表格多、排版复杂、可能含页眉页脚噪音

## 13.2 验证步骤

1. 用旧解析方案处理困难文档
2. 记录解析与问答结果
3. 用增强方案重新处理同一批文档
4. 对比抽取质量、chunk 质量、引用可读性和问答效果
5. 记录剩余失败类型

## 13.3 重点比较指标

- 困难 PDF 导入成功率
- 引用可读性
- chunk 语义完整度
- 问答表现改进情况
- 解析异常是否可定位

## 14. 本期结束后的产品状态

Phase 3 完成后，产品应达到：

- 已经不仅能处理简单文档，还能逐步吸收复杂资料
- 用户能理解为何某些文档效果差，以及如何继续优化
- 上游数据质量不再成为完全不可控的黑盒问题

## 15. 进入 Phase 4 的条件

当以下条件满足时进入 Phase 4：

- 系统已具备相对稳定的导入、检索、回答主链路
- 文档解析能力已经能支撑日常使用
- 用户开始更关注如何高频使用、更新和维护系统
