# Phase 5 PRD：评测与稳态建设

## 1. 阶段目标

本阶段的目标是为系统建立一套可重复执行的评测与回归检查机制，让产品后续迭代不再依赖主观感觉，而有稳定的质量判断标准。

一句话目标：

**让系统每次优化后都能回答一个问题：到底是变好了，还是只是看起来变好了。**

## 2. 为什么现在做这一期

在前四期之后，系统应该已经具备：

- 可用的文档问答能力
- 一定的检索质量优化
- 更好的文档进入系统能力
- 可重复使用的日常流程

此时最大的风险会变成：

- 改一点参数，不知道结果有没有变好
- 换一个策略，新的问题又引入了新的退化
- 某次优化让一类问题变好，另一类问题变差，但没人及时发现

因此本期要建立的是“长期可迭代的质量保障能力”。

## 3. 用户问题

本期主要解决以下问题：

- 系统质量无法稳定评估
- 功能迭代后可能悄悄退化
- 没有标准问题集和统一对比方式
- 无法回答“这次改动值不值得保留”
- 发布和阶段收尾缺少质量门槛

## 4. 本期范围

本期建议覆盖以下能力：

1. 标准问题集定义
2. 问题分类和标注规范
3. 回答与检索结果评估方式
4. baseline 对比流程
5. 回归检查清单
6. 发布前质量检查模板

## 5. 本期不做

本期明确不做：

- 企业级自动标注平台
- 复杂在线实验平台
- 大规模人工标注运营体系
- 完整 A/B 实验系统
- 面向海量团队成员的质量平台

## 6. 用户流程

### 流程 1：新版本验证

1. 用户完成一项功能优化
2. 用户运行既定问题集
3. 系统记录检索与回答结果
4. 用户与 baseline 做对比
5. 判断本次改动是否保留

### 流程 2：回归检查

1. 用户对核心链路做改动
2. 用户执行回归问题集
3. 发现某类问题命中率下降
4. 用户决定回滚、继续修复或接受取舍

### 流程 3：阶段收尾检查

1. 用户准备结束某一阶段开发
2. 用户执行发布前检查清单
3. 确认主要验收项和质量门槛满足
4. 记录已知限制与后续事项

## 7. 功能需求

## 7.1 标准问题集

### 要求

- 定义一套稳定的问题集
- 覆盖核心使用场景
- 支持按类型分类

### 建议分类

- 事实问题
- 多段综合问题
- 术语解释问题
- 困难检索问题
- 无依据拒答问题
- 复杂文档问题

### 验收点

- 问题集可复用
- 问题集能覆盖主要产品能力

## 7.2 结果记录规范

### 要求

- 记录问题、回答、引用、命中来源和评审结果
- 可区分检索问题与回答问题
- 可对比不同版本结果

### 验收点

- 不同版本测试结果具备可比性
- 出现问题时能追踪到具体题目和类别

## 7.3 评估规则

### 要求

对每个问题至少评估：

- 是否检索到正确来源
- 回答是否基于证据
- 回答是否足够完整
- 引用是否可用
- 无依据时是否正确拒答

### 验收点

- 人工评估标准一致
- 结果不再完全依赖主观印象

## 7.4 baseline 对比流程

### 要求

- 保留一个稳定 baseline
- 新方案必须与 baseline 对比
- 比较结果应覆盖命中、回答质量和引用表现

### 验收点

- 每次重要优化都有前后对比结果
- 能明确判断是整体提升还是局部取舍

## 7.5 回归检查机制

### 要求

- 定义哪些问题属于核心回归检查集
- 每次关键改动后必须检查
- 出现回归时有记录与处理建议

### 验收点

- 关键能力退化能够被及时发现
- 系统不会在频繁迭代中悄悄失去原本能力

## 7.6 发布前检查模板

### 要求

- 在阶段性完成时执行统一检查
- 包含功能、质量、已知问题和文档更新项

### 验收点

- 每一阶段收尾时都有一致的质量判断方式

## 8. 非功能需求

## 8.1 可重复性

同一问题集、同一版本、同一配置下，结果评估过程应尽量一致。

## 8.2 可追踪性

每个结论应能追溯到具体问题、具体回答和具体引用。

## 8.3 低门槛

本期的评测体系必须适合当前项目规模，不能因为过度复杂而没人真正执行。

## 9. 数据与元数据要求

本期应重点沉淀：

- question_id
- 问题类别
- 参考文档 / 参考证据
- 当前版本回答
- 当前版本引用
- 检索命中结果
- 人工评估结果
- 版本号 / 日期 / 配置摘要

## 10. 风险与取舍

## 风险 1：评测体系设计过重

### 说明

如果把评测体系做成复杂平台，当前项目很可能无法落地执行。

### 取舍

优先采用轻量、可重复、适合个人项目的方式，先做人能长期坚持执行的体系。

## 风险 2：问题集不够真实

### 说明

如果问题集太偏演示，会让评测失真。

### 取舍

问题集应尽量来源于真实使用场景和真实失败案例。

## 风险 3：只看总分，不看细分类别

### 说明

总分提升可能掩盖局部退化。

### 取舍

必须按问题类型分类看结果，尤其是拒答类、困难检索类和复杂文档类。

## 11. 依赖项

本期依赖：

- 前几期已有相对稳定主链路
- 已有一组真实使用问题或失败案例
- 已有至少一个 baseline 可比较

## 12. 验收标准

满足以下条件可视为本期完成：

1. 已建立标准问题集和问题分类
2. 已建立统一的结果记录格式
3. 已建立至少一套人工评估规则
4. 重要改动可与 baseline 做对比
5. 已具备阶段收尾检查清单

## 13. 验证方法

## 13.1 问题集验证

检查问题集是否：

- 覆盖主要场景
- 含有一定难度层次
- 包含无依据问题
- 包含复杂文档问题

## 13.2 评估流程验证

检查是否可以对一次改动执行完整流程：

1. 跑问题集
2. 记录结果
3. 做人工评审
4. 与 baseline 比较
5. 输出结论

## 13.3 回归验证

选择一次真实改动，验证系统是否能通过问题集发现变好或变差的地方。

## 14. 本期结束后的产品状态

Phase 5 完成后，产品应达到：

- 不再依赖感觉判断质量
- 每次重要改动都可追踪效果变化
- 系统具备长期迭代的基本稳态能力
- 未来继续加功能时，有质量护栏可依赖

## 15. 本期完成后的长期价值

这一期完成后，整个系统会从“持续演化的个人项目”升级为“有产品治理意识的本地知识系统”。

这意味着后续你不只是会继续做功能，还会知道：

- 哪些改动真的有效
- 哪些优化只是看起来高级
- 哪些问题值得保留进入下一阶段路线图
