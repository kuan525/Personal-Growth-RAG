# Phase 2 PRD：检索质量提升

## 1. 阶段目标

本阶段的目标是提升系统检索到正确上下文的能力，让用户更少遇到“资料里明明有答案，但系统没找到”或“找到了相关内容，但不够准”的问题。

一句话目标：

**把 Phase 1 的“能用”提升为“更可靠地找到对的内容再回答”。**

## 2. 为什么现在做这一期

当 Phase 1 完成后，系统已经具备最小可用价值，接下来最常见的痛点通常不再是“系统跑不起来”，而是“检索结果不够好”。

如果不先解决检索质量问题：

- 回答质量上限会被严重限制
- 用户容易误以为模型不行，实际上问题出在召回
- 后续解析和可用性建设的收益也会被削弱

因此，第二阶段优先解决召回与排序问题，是最符合产品收益的下一步。

## 3. 用户问题

本期主要解决以下问题：

- 文档中有答案，但没被召回
- 召回结果语义相关但不够精确
- top_k 中噪音过大，影响回答质量
- 多段信息问题缺少完整上下文
- 不同检索参数下系统表现差异大，缺少对比依据

## 4. 本期范围

本期建议覆盖以下能力：

1. 在现有检索链路上引入混合召回思路
2. 支持 dense retrieval 与 sparse retrieval 的组合
3. 引入 rerank 机制提升上下文排序质量
4. 支持基础检索参数调优
5. 支持 baseline 与增强链路对比验证
6. 对失败案例形成明确分类

## 5. 本期不做

本期明确不做：

- 大规模文档解析重构
- OCR 强化与视觉解析
- 多索引治理平台
- 自动调参系统
- 复杂 Agent 检索策略
- 企业级在线评测平台

## 6. 用户流程

### 流程 1：用户提出问题

1. 用户输入问题
2. 系统执行向量召回、关键词召回或二者组合
3. 系统汇总候选上下文
4. 系统执行 rerank
5. 系统选择更优上下文传给回答模块
6. 系统返回回答和引用

### 流程 2：对比基线效果

1. 用户选择或记录同一问题集
2. 系统分别以 Phase 1 baseline 和 Phase 2 增强模式执行
3. 对比命中结果、引用质量和回答效果
4. 根据结果确定默认方案

## 7. 功能需求

## 7.1 混合召回能力

### 要求

- 在现有 dense retrieval 基础上增加 sparse / BM25 思路
- 支持两类结果合并
- 能明确知道最终上下文来自哪些召回路径

### 价值

有助于补足纯向量召回对关键词、术语、精确匹配场景的不足。

### 验收点

- 同一问题可尝试不同召回模式
- 对关键词明显的问题，召回质量有改善

## 7.2 rerank 能力

### 要求

- 对初始召回结果进行二次排序
- rerank 后能输出更接近问题真实需求的上下文
- 支持基础参数配置

### 价值

有助于降低 top_k 噪音，提升最终上下文质量。

### 验收点

- rerank 后前几条上下文更接近答案来源
- 回答质量比未 rerank 时更稳定

## 7.3 检索参数调优

### 要求

- 支持调节基础 top_k
- 支持区分召回阶段 top_k 与 rerank 后 top_k
- 记录不同参数下的主要表现差异

### 验收点

- 不同参数组合可被比较
- 能形成一套默认参数建议

## 7.4 基线对比能力

### 要求

- 保留 Phase 1 baseline 作为比较对象
- 能在同一问题集上做增强前后比较
- 记录变化结果

### 验收点

- 能明确知道 Phase 2 是否真的优于 Phase 1
- 不能只凭主观感觉判断提升

## 8. 非功能需求

## 8.1 可解释性

检索链路虽然更复杂，但仍应让用户理解：

- 召回来源有哪些
- rerank 是否生效
- 最终回答使用了哪些上下文

## 8.2 可调试性

遇到效果不佳时，应尽量能分辨问题来自：

- dense retrieval
- sparse retrieval
- 合并策略
- rerank
- 回答阶段

## 8.3 性能可接受

本期允许增加一定延迟，但不应把问答体验拖到不可接受程度。若延迟变大，必须明确记录是为了换取更高质量。

## 9. 数据与元数据要求

在 Phase 1 的基础上，本期应进一步支持：

- 检索来源标记
- 初始召回顺序信息
- rerank 后顺序信息
- 问题与命中结果的对比记录

这些数据用于分析增强是否真实有效。

## 10. 风险与取舍

## 风险 1：链路更复杂，调试难度提高

### 说明

引入 hybrid retrieval 和 rerank 后，效果提升不一定线性，问题定位也会变难。

### 取舍

保持每个阶段可观测，尽量不要一次加太多策略。

## 风险 2：质量提升不明显

### 说明

如果评测集太简单，可能看不出 Phase 2 的价值。

### 取舍
n
需要补充更有挑战的问题集，并以“文档里有但常常漏掉”的问题为重点。

## 风险 3：延迟增加

### 说明

多阶段检索和 rerank 会带来额外成本。

### 取舍

本期接受一定延迟，只要质量收益明显且在本地使用场景中仍可接受。

## 11. 依赖项

本期依赖以下前提：

- Phase 1 已经完成
- 已经有一组真实问题集
- 已经记录了部分 baseline 失败案例
- 已有基础索引和问答链路可复用

## 12. 验收标准

满足以下条件可视为本期完成：

1. 系统具备至少一种混合召回方案
2. 系统具备 rerank 能力
3. 在同一问题集上可对比 baseline 与增强结果
4. 已知答案命中率或困难问题表现明显优于 Phase 1
5. 失败案例有明确分类，而不是只知道“效果不好”

## 13. 验证方法

## 13.1 问题集要求

建议在 Phase 1 问题集基础上扩充：

- 关键词明显的问题
- 多段综合问题
- 容易检索偏移的问题
- 术语精确匹配问题

## 13.2 验证步骤

1. 用 baseline 跑完整问题集
2. 记录命中情况与回答情况
3. 用 Phase 2 增强方案跑同一问题集
4. 比较两组结果
5. 分类总结提升点与失败点

## 13.3 重点比较指标

- 已知答案命中率
- 困难问题改善比例
- 引用质量
- 回答是否更少偏题
- 延迟变化是否可接受

## 14. 本期结束后的产品状态

Phase 2 完成后，产品应达到：

- 不只是“能回答”，而是更稳定地找到对的证据再回答
- 用户能感知到系统在真实问题上的命中率提升
- 检索问题开始可被结构化分析和优化

## 15. 进入 Phase 3 的条件

当以下情况成立时，可进入 Phase 3：

- 检索策略已经有基础可用方案
- 剩余主要问题越来越多来自文档解析质量而不是召回策略本身
- 用户已经开始接触结构复杂、表格多、版式差异大的文档
