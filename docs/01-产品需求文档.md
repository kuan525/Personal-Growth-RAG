# 产品需求文档

## 1. 问题定义

你现在面临的问题不是“没有资料”，而是资料过于分散：文档、笔记、项目计划、聊天总结、复盘、决策记录、学习材料都散落在不同位置。

普通搜索只能找文件；通用 LLM 不知道你的历史；简单 RAG demo 只能回答文档内容，但不能持续成长为个人决策支持系统。

本产品要解决的问题是：

> 把分散的个人信息，转化成一个可持续增长、可引用、可检索、可关联、可辅助决策的个人知识系统。

## 2. 目标用户

当前目标用户就是你自己：一个希望通过真实项目掌握 RAG，并最终构建长期可用个人知识系统的开发者和学习者。

系统优先服务：

- 真实个人使用
- 项目学习
- 实现推进
- 阶段复盘
- 决策辅助
- 长期知识积累

系统不优先服务：

- 企业知识库
- 团队协作
- 泛化 demo
- SaaS 产品

## 3. 核心用户场景

### 场景 A：文档问答

用户导入文档后提出事实或概念问题。

期望结果：

- 回答基于 retrieved chunks
- 带 citation
- 能看到 source file
- evidence 不足时说明不确定或拒答

### 场景 B：项目上下文检索

用户询问某个正在进行或过去做过的项目。

期望结果：

- 检索相关文档和笔记
- 找到过去的 decision
- 识别未解决 problem
- evidence 足够时给出 next action

### 场景 C：增量维护知识库

用户新增、修改或删除资料。

期望结果：

- 只处理变化内容
- 旧 chunk 被 invalidated
- 被删除的 document 不再被 citation
- index 状态可查看

### 场景 D：个人图谱查询

用户提出关系型问题。

期望结果：

- project、skill、goal、decision、problem、resource、evidence 能被关联
- graph relation 有 source chunk 支持
- 错误 relation 可以修正

### 场景 E：复盘与成长分析

用户导入 weekly review、project review 或 milestone review。

期望结果：

- 自动提取 goal、progress、blocker、lesson、next action
- 检测 repeated problem
- 生成带 citation 的时间段总结

### 场景 F：决策辅助

用户在某个实践或时间节点提出决策问题。

期望结果：

- 检索相关历史
- 比较 options
- 给出 recommendation
- 展示 evidence、risk、missing information
- 记录 decision，后续可 review outcome

## 4. 功能需求

### FR-1：Source Ingestion

系统必须支持导入 PDF、Markdown、TXT、项目笔记、review note 和 decision log。

验收标准：

- 支持类型能被发现并处理
- 不支持类型有明确提示
- 每个 source 都有稳定 metadata

### FR-2：Chunking and Metadata

系统必须把内容切成可检索的 chunk，并保留稳定 ID 和 source metadata。

验收标准：

- 每个 chunk 关联 document
- chunk order 保留
- citation 能回到 source location

### FR-3：Retrieval and Answering

系统必须检索相关 chunk，并基于 evidence 回答问题。

验收标准：

- 事实问题能检索到正确 chunk
- 回答包含 citation
- evidence 不足时拒答或降级回答

### FR-4：Incremental Update

系统必须识别 new、modified、deleted、unchanged document。

验收标准：

- 修改后的内容替代旧 active chunk
- 删除的 document 不参与默认 retrieval
- unchanged document 不重复处理

### FR-5：Evaluation

系统必须能运行稳定问题集并记录结果。

验收标准：

- 重要改动可与 baseline 对比
- failure 可以分类
- regression 可见

### FR-6：Personal Graph

系统必须抽取并管理高价值 personal entities 和 relations。

验收标准：

- entity 有 type 和 evidence
- relation 有 evidence chunk
- graph output 可查看、可修正

### FR-7：Memory

系统必须把部分高价值知识沉淀为长期 memory。

验收标准：

- memory 有 source、type、timestamp、status
- memory 可修正或 invalidated
- 临时 fact 不会被静默当成长期事实

### FR-8：Decision Support

系统必须支持结构化 decision question。

验收标准：

- 适用时至少比较两个 options
- 区分 evidence 和 inference
- 列出 risk 和 missing information
- decision 可保存并后续 review

## 5. 非功能需求

### NFR-1：Traceability

每个 answer、graph relation、memory、recommendation 都必须能追溯到 source evidence。

### NFR-2：Local-first

系统默认本地运行，单用户使用。

### NFR-3：Explainability

用户应该能理解系统为什么这么回答或推荐。

### NFR-4：Correctability

错误的 relation、memory、document state 必须能修正。

### NFR-5：Practicality

每个阶段都必须交付真实可用能力，而不是只交付 infrastructure。

## 6. 成功指标

### MVP 指标

- document ingestion success rate
- citation coverage
- factual answer acceptance rate
- refusal correctness

### Retrieval 指标

- known-answer hit rate
- difficult-question improvement rate
- citation usefulness
- latency impact

### Incremental 指标

- new document update success
- modified document replacement correctness
- deleted document invalidation correctness
- index state visibility

### Graph 指标

- entity extraction usefulness
- relation evidence coverage
- graph-assisted answer improvement
- correction workflow success

### Decision 指标

- recommendation evidence coverage
- option comparison quality
- risk identification quality
- follow-up review completion

## 7. 硬性否决标准

出现以下情况，产品不能算合格：

- answer 没有 citation
- deleted document 仍然被引用
- recommendation 没有个人历史 evidence
- graph relation 没有 source
- memory 无法修正
- 改动无法通过 baseline 评估
