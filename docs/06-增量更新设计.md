# 增量更新设计

## 1. 设计目标

Incremental update 是 RAG demo 和长期个人知识系统之间的关键区别。

系统必须能处理不断变化的 sources，避免每次 full rebuild，也避免 stale citation。

## 2. 受更新影响的对象

incremental update 最终需要覆盖：

- documents
- blocks
- chunks
- embeddings
- vector index
- sparse index
- entity mentions
- graph relations
- memories
- evaluation state

## 3. 文件状态检测

每个 source file 需要跟踪：

```text
source_path
file_size
last_modified_at
content_hash
document_id
last_ingested_at
status
```

检测状态：

```text
new
unchanged
modified
deleted
failed
```

## 4. Document 状态模型

document status：

```text
active
modified
deleted
failed
superseded
```

chunk status：

```text
active
superseded
deleted
```

默认 retrieval 只能使用 active chunks。

## 5. New Document Flow

1. 检测 file 为 new。
2. 创建 document_id。
3. Parse source。
4. 创建 blocks。
5. 创建 chunks。
6. 存储 chunk metadata。
7. 生成 embeddings。
8. 插入 vector index。
9. 插入 sparse index。
10. 如果启用 graph，则抽取 graph candidates。
11. 标记 document active。

## 6. Modified Document Flow

1. 检测 content_hash 改变。
2. 保持 document_id，除非版本策略要求新版本。
3. 将已有 active chunks 标记为 superseded。
4. 禁用旧 embeddings 的 retrieval。
5. Parse new content。
6. 创建 new blocks 和 new chunks。
7. 生成 new embeddings。
8. 更新 vector 和 sparse indexes。
9. 删除或 invalidated 该 document 的旧 entity mentions。
10. 重新抽取 graph candidates。
11. 检查受影响 memories。
12. 标记 document active。

## 7. Deleted Document Flow

1. 检测 source file 缺失。
2. 标记 document deleted。
3. 标记 chunks deleted。
4. 禁用 embeddings。
5. 从 sparse index 删除或标记 inactive。
6. 依赖该 evidence 的 relations 标记 inactive。
7. 只依赖该 source 的 memories 标记 needs_review。
8. 确保默认 retrieval 排除 deleted content。

## 8. Unchanged Document Flow

1. 检测 hash unchanged。
2. 跳过 parsing。
3. 跳过 embedding。
4. 保留 active index entries。
5. 记录 scan result。

## 9. Versioning Strategy

初始策略：

- 同一 source_path 保持同一 document_id
- modified content 创建 new chunks
- old chunks 变为 superseded
- 默认 retrieval 只使用 active chunks

未来策略：

- 重要 decision / review documents 可以保留历史版本
- historical versions 支持 timeline 和 decision review
- historical chunks 默认不参与 factual QA，除非显式指定

## 10. Index Update Rules

### Vector index

如果使用 FAISS：

- 可能需要 rebuild 或 tombstone filtering
- metadata store 必须判断 active status

如果使用 Qdrant：

- 使用 payload filters
- 按 chunk_id delete 或 update points

### Sparse index

必须支持：

- insert new chunks
- remove / disable old chunks
- filter by active status

## 11. Graph Update Rules

document 变化时：

- 删除旧 superseded chunks 的 entity mentions
- 从 new chunks 重新抽取 mentions
- 更新 relation evidence
- 没有 active evidence 的 relations 变为 inactive

如果一个 graph relation 的所有 evidence 都 deleted 或 superseded，它不能继续 active。

## 12. Memory Update Rules

source evidence 变化时：

- confirmed memories 不应该被静默删除
- 如果 source 被 deleted 或 superseded，memory 应标记 needs_review
- 用户可以 confirm、edit 或 remove

## 13. Status Report

系统应提供 status 命令，展示：

```text
total_documents
active_documents
modified_documents
deleted_documents
failed_documents
active_chunks
superseded_chunks
last_scan_time
last_index_update_time
```

## 14. 必须测试的场景

1. Add new document。
2. Modify existing document。
3. Delete document。
4. Leave document unchanged。
5. Modify one document and verify unrelated queries still work。

## 15. 硬性失败条件

update system 出现以下情况即失败：

- modified 后旧 fact 仍 active
- deleted document 仍被 citation
- unchanged document 被反复 reprocess
- graph relation 指向 only-deleted evidence
- memory 无法提示 source evidence 已 stale
