# 校验方案

## 1. 校验哲学

一个阶段不是因为“代码写完了”就完成，而是因为它在真实数据上通过了预先定义的 validation。

每次 validation 必须回答：

1. 测试了什么能力？
2. 使用了什么数据？
3. 预期结果是什么？
4. 实际发生了什么？
5. 失败在哪里？
6. 项目应该继续、修复还是回退？

## 2. 校验资产

项目应长期维护以下资产：

- validation question set
- baseline run results
- regression question subset
- failure log
- release checklist
- decision review records

## 3. 问题类型

validation set 必须包含：

```text
fact_qa
concept_explanation
multi_chunk_synthesis
difficult_retrieval
unsupported_question
project_context
graph_query
review_summary
decision_support
```

## 4. Phase 0 校验：基础数据集

### 测试目标

验证实现前是否已有真实 validation data。

### 必须证据

- real documents
- annotated questions
- expected sources
- refusal expectations
- review / decision samples

### 通过标准

- 至少 20 个问题
- 每个问题有 type
- factual questions 有 expected source
- unsupported questions 有 expected refusal behavior
- 至少 3 个问题覆盖 project / review / decision 场景

### 失败标准

- 问题是泛 demo 问题
- 没有 unsupported question
- 没有 source annotation
- 没有个人或项目样本数据

## 5. Phase 1 校验：Local RAG MVP

### 测试目标

验证端到端 citation QA。

### 测试步骤

1. Ingest PDF / Markdown / TXT。
2. Build vector index。
3. 运行 validation question set。
4. 记录 retrieved chunks、answer、citations。
5. 人工判断每个 answer。

### 通过标准

- 所有支持文件类型可以 ingestion
- 至少 90% source files 生成 chunks
- factual QA acceptance 至少 60%
- citation coverage 至少 80%
- unsupported questions 最多 1 个明显 hallucination
- 每个 answer trace 被存储

### 失败标准

- 没有 citation output
- citation 无法追溯到 source chunk
- unsupported questions 被自信编造
- 没有 query log

## 6. Phase 2 校验：Retrieval Quality

### 测试目标

验证 hybrid retrieval 和 rerank 是否真正改善 evidence retrieval。

### 测试步骤

1. 运行 Phase 1 baseline。
2. 使用同一问题运行 enhanced retrieval。
3. 对比 known-answer hit rate。
4. 对比 difficult retrieval questions。
5. 对比 citation usefulness。
6. 对比 latency。

### 通过标准

- known-answer hit rate 提升
- difficult retrieval 至少改善 30%
- citation quality 不退化
- unsupported question behavior 不退化
- retrieval source 和 ranks 被记录

### 失败标准

- 改善只靠主观感觉
- rerank 结果不可检查
- latency 增加但质量无收益
- retrieval 引入更多噪音并导致 hallucination

## 7. Phase 3 校验：Structured Ingestion

### 测试目标

验证 parsing 和 chunking 是否改善输入质量。

### 测试数据

至少包括：

- 一份有 heading 的长 Markdown
- 一份有 page reference 的 PDF
- 一份 difficult PDF 或结构复杂文档

### 测试步骤

1. 运行旧 parser。
2. 运行 structured parser。
3. 对比 chunks 和 metadata。
4. 运行同一问题集。
5. 对比 citation readability。

### 通过标准

- heading 进入 metadata
- PDF page info 在可用时进入 citation
- chunk boundaries 改善
- difficult document QA 改善
- parse errors 被分类

### 失败标准

- 新 parser 存了 metadata，但 retrieval / citation 不使用
- difficult documents 没有可见改善
- 失败仍然不可诊断

## 8. Phase 4 校验：Incremental Update

### 测试目标

验证 add / update / delete 行为安全。

### Test A：Add Document

预期：

- new document 被检测
- 只处理 new document
- new content 可 searchable

### Test B：Modify Document

预期：

- changed hash 被检测
- old chunks 被 superseded
- new answer 使用 new content
- old content 不被 citation

### Test C：Delete Document

预期：

- document 标记为 deleted
- chunks disabled 或 deleted
- retrieval 不引用 deleted content

### Test D：Unchanged Document

预期：

- unchanged document 被跳过
- unrelated answers 不退化

### 通过标准

- 四组测试全部通过
- update logs 存在
- status report 准确

### 失败标准

- deleted document 仍出现在 citation
- modified document 返回旧事实
- update process 破坏无关 document
- state 不可检查

## 9. Phase 5 校验：Personal Graph

### 测试目标

验证 graph relation 是否有用且有 evidence。

### 测试步骤

1. 从 project / review / decision documents 中抽取 entities。
2. 抽取 relations。
3. 检查 graph entries。
4. 运行 graph questions。
5. 对比 graph-assisted retrieval 和 non-graph retrieval。

### 通过标准

- 每条 relation 有 evidence_chunk_id
- 高价值 entity types 被抽取
- graph entries 可检查
- wrong entries 可 disabled
- 至少 3 个真实问题因 graph context 改善

### 失败标准

- graph 没有 evidence
- graph output 不可修正
- graph 没有改善任何 user scenario
- extraction 产生大量低价值噪音 entity

## 10. Phase 6 校验：Memory and Review

### 测试目标

验证 review 是否能变成有用长期上下文。

### 测试步骤

1. Ingest 多份 review notes。
2. 抽取 goals、problems、lessons、next actions。
3. 生成 time-based summary。
4. 提升 selected memory candidates。
5. 查询 repeated problems 和 project progress。

### 通过标准

- extracted memories 有 sources
- summaries 引用 review records
- repeated problems 可检测
- memory 可 confirmed / rejected / superseded

### 失败标准

- summary 泛泛而谈且无 citation
- temporary notes 自动成为 permanent memories
- memory 没有 source
- time filters 不工作

## 11. Phase 7 校验：Decision Support

### 测试目标

验证 recommendation 是否基于个人历史。

### 测试步骤

1. 提出 decision-support questions。
2. 检查 retrieved history。
3. 检查 option comparison。
4. 检查 evidence / inference separation。
5. 保存 decision。
6. 后续 review outcome。

### 通过标准

- recommendation 引用 personal evidence
- 适用时至少比较两个 options
- risk 和 missing information 被列出
- unsupported decision 触发 clarification
- decision 被保存以供 future review

### 失败标准

- recommendation 是泛泛建议
- 没有 citation
- uncertainty 被隐藏
- decision 无法后续 review

## 12. Phase 8 校验：Operational Hardening

### 测试目标

验证质量可以长期维护。

### 通过标准

- regression subset 存在
- 每次重要改动可 evaluation
- baseline comparison 可执行
- failure categories 被跟踪
- release checklist 被使用

### 失败标准

- 没有可重复 evaluation
- 没有 configuration record
- regression 无法发现

## 13. 人工评估 Rubric

每个 answer 从以下维度评估：

```text
source_hit: yes / partial / no
groundedness: strong / partial / weak / none
completeness: complete / partial / missing
citation_quality: useful / weak / wrong / none
refusal_quality: correct / wrong / over-refused / not_applicable
overall: pass / partial / fail
```

## 14. Release Gate

一个 phase 只有满足以下条件才能关闭：

- pass criteria 达成
- failed cases 被记录
- non-goals 没有被误实现
- next phase inputs 准备好
- known limitations 明确写出
